# 007 — Musical effects: pitch, time, tempo-synced echo, crush

**Status:** Draft
**Date:** 2026-07-23
**Issues:** (filled by /spec-to-issues)

## Summary

Extends spec 004's per-track effects with the musical properties the product's psychoacoustics principle says come first: **pitch** (shift in semitones), **musical time** (stretch by musical ratios), **timbre** (a fourth one-knob Crush macro), and **tempo** (a per-track BPM estimate that lets Echo sync its delay time to the track's beat). Pitch and stretch are *source transforms* — high-quality offline renders that replace the track's playing buffer — not insert effects; the spectrogram responds the way hearing does: pitch shift translates the CQT image vertically (log-frequency makes this exact), stretch scales it in time, and the melody overlay's note blocks move instantly because they transform mathematically. Everything manipulates tracks in their entirety: note-level editing of timbre/pitch inside a mix is explicitly out of scope.

## Goals

1. **Pitch shift.** Each track has a pitch offset in semitones, −12…+12, adjusted by stepper (tap = ±1). On commit the track's audio re-renders offline through the selected stretch/shift library (duration preserved), the playing buffer swaps seamlessly, and the spectrogram re-analyses — the 440 Hz fixture shifted +12 peaks exactly 24 CQT bins higher. The melody overlay's notes shift by the same semitones *immediately* (mathematical transform), before the audio render lands.
2. **Time stretch.** Each track has a speed ratio from {½×, ¾×, 1×, 1½×, 2×} (pitch preserved). Duration, `totalTime`, and scroll range adapt; melody note times scale by 1/ratio immediately; spectrogram re-analyses at the new duration.
3. **Crush.** A fourth one-knob macro (bit-depth reduction, `Tone.BitCrusher`) joins the chain ahead of Space/Echo/Tone, with the same live-audio + preview-overlay + debounced-tile-refresh behavior as the existing three.
4. **Tempo estimate.** Every track gets a BPM estimate + confidence (essentia `RhythmExtractor2013`), computed once in a worker at upload/restore-without-estimate time, persisted per track. Estimation failure or low confidence is a first-class state, not an error.
5. **Tempo-synced Echo.** When the active track has a confident tempo, the Echo control gains subdivision choices (¼, dotted ⅛, ⅛, ⅛ triplet) that set `delayTime = subdivision × 60/BPM`; the estimated BPM is visible in the drawer. No confident tempo → Echo keeps today's fixed 0.25 s and the subdivision UI is absent (not disabled-and-mysterious). Live chain, preview render, and offline tile render all use the same per-track delay time.
6. **Persistence, undo, integrity.** All new params (pitch, stretch, crush, echo sync, tempo estimate) persist with the project, restore on load, participate in undo/redo as single-gesture dispatches, and survive channel recreation (#212 class). Existing projects' persisted spectrograms do **not** spuriously re-analyse after the hash format grows new fields (legacy-hash migration).
7. **Existing contracts unchanged.** Tap toggles playback, drag seeks, recording works, loudness normalization still applies, and spec 004's effects behave exactly as before when the new params are at defaults.

## Non-goals

- **Note-level manipulation** (changing timbre/pitch/duration of individual notes inside a track). The owner's own framing: too ambitious for this app — we manipulate tracks in their entirety according to perceptual categories. Revisiting requires source separation or MIDI-domain editing (#182's world), not this pipeline.
- **Spatial panning** — deferred to a later spec (owner direction, this session).
- **Beat grid, onset/tempo visualization on the timeline or playhead, quantized editing** — #165's scope, deliberately untouched. The tempo estimate this spec ships is #165's future input, not its delivery.
- **Continuous pitch (cents/detune) and free-ratio stretch.** Nuance-layer controls; the categorization layer ships first. The ratio set keeps rhythmic relationships musically legible (half-time/double-time) instead of silently desynchronizing a track from the mix by 7%.
- **Project-tempo conform** ("stretch all tracks to a shared tempo") — needs beat *positions* and a product decision about what a project tempo is; flagged as the natural sequel.
- **Formant preservation / vocal-quality modes** — library-dependent polish; human QA decides if the default artifacts are acceptable before more knobs are added.
- **Live granular preview while adjusting pitch/stretch** (e.g. a temporary `Tone.PitchShift` insert during interaction). Discrete steppers + instant melody/tile response + debounced render make it unnecessary in v1; reconsider only with QA evidence of felt latency.
- **Re-transcription after transform.** Melody data transforms mathematically (exact for semitones/ratios); re-running Basic Pitch would cost seconds to recompute what arithmetic gives for free.
- **Persisting the transformed audio buffer.** V1 re-renders the transform on project load when params are non-default (originals stay the single audio source of truth in `audioData`); a profiling issue gates whether a derived-buffer store (which must join `TRACK_DATA_STORES`) is warranted.

## Design

### Council decision record

> **Decision 1 — pitch shift and time stretch are offline source transforms, not real-time insert nodes.** A new transform stage sits upstream of the mixer chain: `originalBuffer → SourceTransform (worker) → playing buffer → Player → EffectsChain → Channel`. On commit, the transform renders in a Web Worker, the track's channel is rebuilt with the new buffer (the existing `recreateChannel` path), and the spectrogram re-analyses through the existing `EffectsRefreshScheduler`.
> **Rationale:** an insert node cannot change duration, so stretch forces this shape anyway; one mechanism for both keeps pitch+stretch a single render (they compose in one library call). Quality: granular real-time shifters (`Tone.PitchShift`, `GrainPlayer.detune`) carry chorus/flutter artifacts the offline libraries avoid. Verifiability decides the remainder: this sandbox never observes live worklet output (`kb/verification.md` #542), while an offline-rendered buffer is falsifiable at unit level with the 12-TET CQT bin test — the strongest verification this repo has.
> **Dissent:** *Performance/Product* warn the render latency (~1–5 s on long tracks, same class as upload analysis) means audio feedback is not instant; accepted because the categorical layers respond instantly (melody notes and a provisional tile translate/scale) and commits are discrete taps, not drag ticks. *Adversary* notes the buffer swap while playing must preserve transport position and effect params — the #212 regression class — and gets a named regression test.

> **Decision 2 — categorical controls: semitone steppers and a ratio set, no 0–100 sliders for pitch/time.** Pitch: `−12…+12` stepper. Stretch: segmented {½×, ¾×, 1×, 1½×, 2×}.
> **Rationale:** this is `kb/product.md`'s two-layers principle applied to input: pitch and musical time are *categorization-layer* properties, manipulated in the units musicians think in. Practically: each tap is one dispatch (undo granularity = gesture granularity for free), the debounced scheduler coalesces rapid taps, and the entire Radix-slider commit/drag-lifecycle gotcha class (CLAUDE.md) is avoided because there is no drag.
> **Dissent:** *Simplicity* asked whether stretch could ship fewer ratios ({½, 1, 2}); kept ¾/1½ because dotted-feel and half-time-shuffle relationships are the musically common cases. *Product* flags the stepper must still feel playful (it's the first non-slider control in the drawer) — human QA checks reach and feel.

> **Decision 3 — transform library: `signalsmith-stretch` (MIT) as primary candidate, selected by an M1 evaluation, `soundtouchjs` (LGPL-2.1) as fallback; `rubberband-web` rejected on license (GPL-2.0+).** The evaluation is a real-shape + quality gate, not a benchmark essay: run the candidate in a worker and in vitest, feed the 440 Hz and burst fixtures, assert the 12-TET bin shift (±2 bins), duration tolerance (±1 hop), finite output, and record processing time; pick the first candidate that passes; capture the decision in `kb/decisions.md`.
> **Rationale:** license first (this is a deployed app), then verifiable quality. Signalsmith Stretch is polyphonic, small, and MIT; its npm packaging shape (worklet-oriented vs. plain WASM API) is exactly the kind of assumption five classification bugs taught us to validate before mocking (`kb/verification.md`).
> **Dissent:** *Adversary* predicted the npm package would be worklet-first — **confirmed by direct package inspection** (see Prior art in Grounding): there is no plain-function API, so the offline render runs the worklet node on an `OfflineAudioContext` (`audioWorklet.addModule` on the offline context, `addBuffers` + `schedule({rate, semitones})` + `start`, then `startRendering`). This merges with the BitCrusher offline-worklet validation into one M1 question: *do worklet nodes process correctly during offline rendering in this app's contexts (and in the sandbox's offline renders — #542 covers live delivery, not offline)?* The quality-gate assertions consequently run browser-side (a `zzz-`/e2e harness feeding results through the CQT analysis), not in vitest. If offline-worklet rendering fails, the fallback-of-last-resort is a plain WSOLA implementation in the worker (quality gate still applies); that outcome would be a spec-level check-in, not a silent substitution.

> **Decision 4 — tempo estimation: essentia `RhythmExtractor2013` (or `PercivalBpmEstimator` if the M1 shape-check favors it), one-shot per track in a worker, output = `{bpm, confidence}` only.** Runs where classification runs (worker + `essentiaLoader` reuse), triggered per track at upload and at restore-when-absent; persisted on the track record.
> **Rationale:** the WASM is already shipped and cached; the algorithm is battle-tested where a hand-rolled onset-autocorrelation would be new DSP to tune and maintain. One-shot per track avoids essentia's per-frame cost class (#236). Scoping to a single number + confidence keeps #165 (beats, grid, visualization) genuinely deferred.
> **Dissent:** *Adversary*: octave errors (2×/½× BPM) are endemic to tempo estimators — partially benign here because subdivision choices absorb a doubling, but the QA checklist includes "does synced Echo feel right on real music", and the UI shows the BPM so a wrong number is visible rather than mysterious. *Simplicity* wanted to skip confidence handling; overruled — rubato/ambient tracks are common amateur material, and "no confident tempo" must degrade to today's behavior, not to a wrong-feeling sync.

> **Decision 5 — Crush joins `EFFECT_ORDER` first (`crush → space → echo → tone`) via `Tone.BitCrusher`; Echo's `delayTime` becomes per-track when synced, set by snap, never ramped.** The macro maps amount → bit depth (16→~3, exponential) plus wet. Extending `EFFECT_ORDER` changes `hashEffectAmounts` — a **legacy-hash migration** normalizes stored 3-field hashes to the new format on comparison so existing projects don't mass re-analyse on first load.
> **Rationale:** distortion-before-space is the conventional and better-sounding order; bit-depth reduction is the timbre/roughness dimension the owner asked for ("bit quantization, downsampling"). For Echo sync, changing `delayTime` while running pitch-warps the echoes (#492's reason for fixing it) — but a *discrete* subdivision commit may snap it (momentary discontinuity on an explicit user action, same license as snap-on-reactivation), which amends the "character params are fixed constants" rule into "character params are fixed *during* interaction; discrete commits may snap them," recorded here deliberately.
> **Dissent:** *Adversary*: `Tone.BitCrusher` is **worklet-based** (`BitCrusher.worklet.ts`) — whether it renders correctly on a fresh manual `Tone.OfflineContext` (renderTrackOffline's shape) is unproven; M1 validates it against real Tone before any mock exists (#489's lesson). If it fails offline, the documented fallbacks are (a) a pure-math crush applied in the render path (quantize samples directly) with the live node kept for real-time — same audible result, two implementations, flagged as a cost — or (b) switching the macro's character to `Tone.Chebyshev` (plain waveshaper, no worklet, works in any context uniformly); which one is decided by ear at the M1 check-in, not silently. *Verification* adds: the sandbox worklet limitation (#542) is about live main-thread delivery, not offline rendering, but that distinction is exactly what the M1 check must confirm empirically.

### Architecture

**Source transform stage** (`features/tracks/SourceTransform.ts` + `sourceTransform.worker.ts`): pure function of `(originalBuffer, pitchSemitones, stretchRatio) → transformedBuffer`, WASM in a worker. `TrackService` gains `applyTransform(trackId, params)`: render → swap buffer into the channel (recreate path, preserving volume/mute/solo/effect signals — the #212 guards already exist) → notify duration change (the `workstationEffects.ts:185` totalTime sync re-runs off the dispatch) → melody data transformed arithmetically and re-saved → spectrogram refresh scheduled. The transform's completion callback checks the track still exists before writing back (CLAUDE.md background-job rule). In-session render results are cached by `(trackId, params)` so undo/redo of a transform is instant.

**One combined "sound hash".** The persisted spectrogram's params hash becomes `pitch:stretch:crush:space:echo:tone:echoSync` (transform + effects + sync). One string, one comparison, one migration point; the legacy 3-field format maps to defaults for the new fields.

**Reducer.** `SET_TRACK_TRANSFORM { trackId, pitchSemitones, stretchRatio }` (one action, both fields — a future combined gesture stays one undo step, per the 2026-07-21 rule) and `SET_TRACK_ECHO_SYNC { trackId, subdivision | null }`; crush reuses `SET_TRACK_EFFECT`. Tempo estimate is written via a `SET_TRACK_TEMPO`-style action from the worker completion (like `SET_INSTRUMENT`), not undoable state the user authored.

**Drawer UI** (`EffectsBottomSheet`): existing three sliders + Crush slider; below them a transform row — Pitch stepper (showing e.g. “+3 st”) and Speed segmented control; Echo gains a subdivision row with a BPM badge when a confident tempo exists. All mobile-first, no hover dependencies.

**Instant categorical feedback.** On transform commit, before the audio render resolves: melody overlay notes shift/scale (arithmetic), and — as an optional milestone-7 enhancement — the preview-overlay registry shows the current tiles translated (pitch: 2 bins/semitone) or time-scaled (stretch) as a provisional image, refined when re-analysis lands. The audio keeps playing the old buffer until the swap; this is live-then-refine inverted (visual leads, audio follows) and is the honest version of it: the categorical layer *is* exactly transformable, the nuance layer needs the render.

## Open questions

1. **Which library passes the M1 gate, and does its npm packaging expose an offline-usable API?** Evidence: the M1 evaluation harness results (bin-shift, duration, artifacts by ear on device, timing). If neither candidate passes, the WSOLA fallback triggers a check-in on this spec.
2. **Control labels and framing** — "Pitch"/"Speed"? “+3 st” vs. “+3”? Half-time/double-time wording? Resolved by human QA on device (checklist), not in code review.
3. **Default subdivision when sync is enabled** — ¼ is safest, dotted ⅛ is the musically iconic delay; QA decides. Also whether synced state should follow BPM re-estimation if the estimate changes after a re-record.
4. **Load-time re-render cost of transformed tracks** — profiling issue decides whether a persisted derived-audio store (added to `TRACK_DATA_STORES`) is warranted.
5. **Cross-track tempo disagreement** — each track syncs to its *own* estimate in v1; whether a project-level tempo should exist (and whether stretch should offer "match track X") is the sequel spec's question, entangled with #165's beat positions.
6. **Varispeed ("tape") link between the two controls.** Koala's Re-pitch mode and Ableton's Repitch warp (Prior art) suggest a coupled pitch↔speed mode is one of the most *intuitive* transforms for casual creators (the turntable/tape metaphor). It falls out of this spec's model nearly for free — a "link" toggle where choosing a ratio also sets `semitones = 12·log₂(ratio)`, and the ratio set maps to genuinely musical intervals (½×/2× = ∓octave; ¾× ≈ −5 st ≈ fourth down; 1½× ≈ +7 st ≈ fifth up) — and could even skip the stretch library entirely (varispeed is plain resampling). Decide during M6 whether it ships as a toggle there or as its own follow-up; if the transform library's offline path proves fragile, varispeed-only is also a graceful degraded mode. Related future note: `signalsmith-stretch`'s node also supports scheduled *live* playback (`rate`/`semitones` changes mid-stream), a possible engine for future scrub-with-pitch or live varispeed — out of scope here, unverifiable live in the sandbox (#542).
