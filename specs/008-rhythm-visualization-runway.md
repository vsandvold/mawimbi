# 008 — Rhythm made visible: onset, tempo, beat, and pulse on the runway

**Status:** Draft
**Date:** 2026-07-23
**Issues:** (filled by /spec-to-issues)

## Summary

Makes rhythm visible on the runway the same way pitch already is — as two distinct perceptual layers. The **categorical layer** (the beat you tap your foot to) renders as thin full-width **beat rungs** laid down where the anchor track's detected beats actually sounded, receding through the memory field with the audio; the **nuance layer** (actual onset timing — swing, push, drag) renders as small per-track **onset ticks** at the runway's rail edges, in track color, whose spatial offsets against the rungs *are* the micro-timing rendering. At the playhead, the loudness-meter frame pulses when a beat crosses the line — the moment of arrival — driven by persisted beat positions and engine time, not live analysis. A cuttable final layer extends the felt pulse through silence: ghosted, visually distinct phantom rungs extrapolated at the induced interval where detection has nothing. Everything degrades honestly: no confident tempo → no rungs, no pulse, no mystery — onset ticks alone.

## Grounding

- **The governing frame** (owner statement, 2026-07-23, `kb/product.md`): the runway is *short-term auditory memory with only the slightest anticipation*. Geometry confirms it in code: future content descends from the horizon (at `playheadFraction − elevationFraction` = 0.15 from the top in the active `noteHighway` preset, `runwayConfig.ts`) through the anticipation strip — the band between horizon and playhead line, ~35% of viewport, perspective-compressed toward the horizon — crosses the playhead line at `playheadFraction × visibleHeight` (0.5), and recedes *below* the line toward the viewer — the memory field is the dominant bottom half (`e2e/track-edit-mode.spec.ts`'s clip math: content at track-time `t` sits at `playheadLineY − (t − now) × pixelsPerSecond`). Beat Saber-style approaching-note mechanics are ruled out by this layout, by design.
- KB `kb/product.md`: **two layers of perception, two renderings** (2026-07-19) — the spectrogram is already rhythm's nuance rendering (transients are visible as broadband edges); what's missing is the categorization layer (beats, meter) and the *relation* between them (micro-timing). **One stream, focusable sources** (2026-07-20) — the induced pulse is a property of the combined stream (one grid), onsets are properties of sources within it (per-track marks); per-track emphasis represents selective attention. Visual identity: the runway rendering *is* the product; no fake pulse decoration — everything rendered must be real analysis output.
- KB `kb/domain.md`: melody note times are **track-buffer-relative** — beat ticks and onset times are too, and every consumer must add `Track.startTime` (the #484 sparkle-burst class: missed once, caught only because an overdub exposes it — a named test with `startTime > 0` is mandatory). Runway geometry facts (nonlinear projection; reduced-motion flattening). `Tone.Transport` is the single clock; `transportTime` is not a clock (CLAUDE.md) — the pulse reads `playback.getEngineTime()`.
- KB `kb/decisions.md`: TimelineRenderLoop partitioning/idle rules (2026-07-22, #541); `BarSmoother`'s reset-on-idle lesson (2026-07-20, #483) — per-frame ballistics state resets at the domain's discontinuity points (`renderLoudnessMeterIdle`), which the pulse envelope inherits; loudness meter replaced the plasma playhead because untestable particle internals killed it (2026-03-15) — every new per-frame element here is a pure, deterministic function of persisted data and engine time; `window.__mawimbi` bridge for data claims (2026-07-20, #480).
- KB `kb/verification.md`: this sandbox **never delivers live worklet frames** (#542) — live-input rhythm visuals are unverifiable here, which is decisive for Decision 4; reduced-motion linear Y mapping (`playheadLineY − (t − now) × pps`) makes mark positions exactly predictable; decoded pixels for paint claims (`e2e/helpers/pixelDecode.ts`); engine-time polling before screenshots, one screenshot for two regions (#484); track colors are deterministic (`COLOR_PALETTE` cycles — don't pin `Math.random` reflexively).

## Goals

1. **Rhythm analysis + persistence.** Every track gets `{ticks: number[], onsets: number[]}` (track-buffer-relative seconds) from one offline worker pass (essentia, sharing #559's `{bpm, confidence}` extraction), computed at upload/restore-when-absent, persisted in a new `rhythms` store that joins `TRACK_DATA_STORES`, restored into the spectrogram cache on load. Low confidence and analysis failure are first-class states (empty ticks, onsets may still exist).
2. **Beat rungs (categorical layer).** When an anchor track exists (Decision 3), thin full-width rungs render across the runway at the anchor's detected tick times — in the memory field *and* the anticipation strip (same honesty as the spectrogram tiles, which already preview the strip) — receding/advancing with the content, correct under zoom (with density limiting), scroll, seek, resize, and reduced motion. No anchor → no rungs, and nothing hints that something is missing.
3. **Onset ticks (nuance layer).** Each track's onsets render as short rail-adjacent marks in the track's color inside that track's own overlay draw (piano-roll path), so they inherit focus/dim/mute treatment for free. A track's swing or push is readable as its ticks' spatial offsets against the rungs — no explicit annotation.
4. **Arrival pulse.** During playback with an anchor, the meter frame pulses (glow/width envelope, instant attack, exponential decay) as each anchor beat crosses the playhead line — driven by persisted ticks + `getEngineTime()`, resetting at every playback discontinuity (the `renderLoudnessMeterIdle` hook point). Supersedes #485's live-flux mechanism for playback.
5. **Pulse continuation (cuttable).** Where the anchor has no detected tick within the anticipation strip (track ended, silence gap) but confidence holds, ghosted phantom rungs extrapolate the induced interval (median of the last k inter-beat intervals, bounded horizon), visually distinct from detected rungs. Never rendered where detected ticks exist; never rendered without a confident anchor.
6. **Transforms and lifecycle stay coherent.** Time stretch (spec 007 #562, when it lands) rescales ticks/onsets by 1/ratio arithmetically like melody times; pitch shift leaves them untouched; deleting a track removes its rhythm row via the shared store list; a track deleted mid-analysis doesn't get an orphan row written back (#540 guard); undo-restore of a track restores its rhythm rendering.
7. **Existing contracts unchanged.** Scrub/seek, tap-to-toggle, recording, zoom, edit mode, and every existing runway invariant suite stay green; with no rhythm data present, rendering is byte-identical to today.

## Non-goals

- **Quantized editing, tempo conform, drum-machine sync** — #165/#25 territory; this spec delivers #165's *visualization* half only (an update comment on #165 re-scopes it).
- **Live-input rhythm visuals** (pulse or marks during recording). The live spectral-flux path (#485's mechanism) is retired for playback and deferred entirely for live input: the sandbox cannot verify it (kb/verification.md #542), and recording-time visuals were already out of scope for sparkles (spec 003). If it returns, it returns as its own spec with its own verification story.
- **Downbeat/bar-line marking.** essentia gives beat ticks, not meter; rendering bar lines would require meter inference we don't have — a heavier rung every 4 beats would be fake precision (kb/product.md: no fake pulse decoration). Revisit only with a real meter-detection source.
- **Explicit micro-timing annotation** (offset whiskers, deviation coloring). The rung-vs-tick geometry already renders swing/push spatially; annotation on top is clutter against "minimal, intuitive." The swung fixture (M1) exists to *prove* the geometry claim; if human QA finds the geometry alone illegible, that evidence reopens this.
- **Per-track beat grids or a grid-source picker UI.** One stream, one induced pulse (Decision 3); v1 has zero new UI controls — rendering appears when data exists, like the piano roll.
- **Beat-synchronized *audio*** (metronome click, tempo-aware count-in) — #199/#165 follow-ups, untouched.
- **Panning/spatial** — still deferred (spec 007 non-goal, owner direction).

## Design

### Council decision record

Inline council (Architect, Adversary, Simplicity, Product, Verification, Performance — the rAF path is touched). No escalation: every disagreement resolved against code or issue evidence.

> **Decision 1 — one worker pass computes `{bpm, confidence, ticks, onsets}`; scalars live on the track record (#559's plan), arrays in a new `rhythms` store joining `TRACK_DATA_STORES`.** The analysis module extends the tempo worker #559 plans (essentia + `essentiaLoader` reuse): `RhythmExtractor2013` already returns `ticks` alongside `bpm`/`confidence` — #559's design throws the beat positions away; this spec persists them (coordination: amend #559 if it lands first, else the shared module ships here and #559 consumes it). Onsets come from a one-shot `SuperFluxExtractor` call in the same pass (`OnsetRate` as fallback — the M1 shape validation picks by output quality on the click fixture, extending spec 007 M1's essentia ritual rather than duplicating it). In memory, rhythm data lives on the spectrogram cache entry beside `melody` (`getRhythm`/`setRhythm` mirroring `getMelody`/`setMelody`), restored per track on project load the way melodies are.
> **Rationale:** ticks are free — the extractor computes them anyway; a second analysis pass later would re-run the whole WASM chain to recover discarded output. Storage: a dense track's onsets run ~5–15/s → ~10–25 KB JSON for a 3-minute track — the same size class as melody data, which got its own store precisely so metadata lists load without bulk (kb/decisions.md 2026-03). Putting arrays on the track record would bloat every project-list load; `rhythms` (keyPath `trackId`, `DB_VERSION` 3→4) joins `TRACK_DATA_STORES` so both delete paths cover it by construction (#540's drift lesson). Transform coherence: ticks/onsets are track-buffer-relative seconds exactly like melody times, so spec 007's stretch milestone scales them by the same arithmetic (`× 1/ratio`) in the same place it scales melody times; pitch shift is a no-op for them.
> **Dissent:** *Simplicity* wanted arrays on the track record (no version bump, no new store) — overruled by the melodies precedent and the #342 multi-tab `blocked` handling already being in place for version bumps. *Adversary* flags octave errors (endemic to tempo estimators) shift every inter-beat interval by 2×/½× — partially benign for rungs (a half-time grid is still *a* truthful pulse reading; tapping half-time is a legitimate percept), visible rather than mysterious since ticks come from real tracked beats, but the QA checklist includes "does the grid feel like the pulse on real music." *Adversary* also notes `TranscriptionService`'s analogous background-save path is still unguarded (CLAUDE.md) — the rhythm write-back must not copy that shape; it checks `spectrogramCache.getEntry(trackId)` before saving, like `extractAndCacheMelody`.

> **Decision 2 — rungs on a dedicated project-level overlay canvas; onset ticks in each track's own overlay draw.** A new `RhythmOverlay` canvas mounts as a sibling of the `.timeline__track` items with an explicit z-index (the CSS-grid stacking-context gotcha; rails own z-index 3, tracks 0–2 — the overlay slots between track content and rails, exact layer a renderer/CSS constant for QA). It registers one normal `TimelineRenderLoop` callback (`peekDirty` on scroll/zoom/data-reference change; plain measure-then-write — no self-affecting reads, so the recording-track exception is explicitly *not* copied). Onset ticks are drawn inside `Spectrogram.tsx`'s existing overlay write beside `writeMelodyOverlay` — short horizontal strokes at both rail-adjacent edges, track color, at each onset's screen Y.
> **Rationale:** the grid is stream-level state — one owner, one canvas — while onsets are source-level, and drawing them in the track's own canvas means focus-lift, edit-mode dimming, and mute styling all apply to them *for free* (the one-stream principle implemented by inheritance rather than new mechanism: attention to a source amplifies its timing marks because it amplifies the whole track layer). Rung-in-anchor-track's-canvas was rejected because the grid would then dim when another track is focused — stream-level state dimming with a source is wrong semantics. Rejected onset placements: full-width per-track lines (clutter destroys the one-stream reading), spectral-centroid x-positioning (false cross-mapping — the x-axis is pitch's; rhythm stays pitch-agnostic), brightening tile columns (decorating the spectrogram violates its nuance-layer honesty — the spectrogram already *is* the transient rendering).
> **Dissent:** *Performance* — marks add O(visible marks) strokes per active frame on canvases that already repaint; the idle skip is untouched, marks are windowed to the canvas window like tiles; no profiling issue warranted unless QA observes jank. *Adversary* — density at `MIN_PIXELS_PER_SECOND` (50 px/s): 120 BPM → rungs every 25 px, dense onsets worse; a level-of-detail rule (drop to every 2nd/4th rung below a named px-spacing threshold; fade ticks below a threshold) is part of Goal 2/3, unit-tested as a pure function. *Product* — default-view tick clutter on dense mixes is the top QA risk; alpha/length/LOD are named renderer constants so tuning never touches tested logic (the spec 003 Q4 pattern).

> **Decision 3 — the grid follows one anchor: the unmuted track with the highest tempo confidence above the shared threshold; ties break to the earliest-created track.** A pure selector `selectRhythmAnchor(tracks, rhythmData) → trackId | null` recomputed on mute/solo/delete/data changes. The confidence threshold is the *same named constant* spec 007's #560 uses to gate echo-sync — one definition of "confident tempo" product-wide. No qualifying track → `null` → no rungs, no pulse, onset ticks only.
> **Rationale:** meter perception induces *one* pulse from the combined stream (kb/product.md); rendering competing grids per track is perceptually false and visually chaotic. Anchoring to the highest-confidence *unmuted* track keeps the grid attached to what is actually audible; other tracks' onset ticks still render, so genuine cross-track tempo disagreement is *visible as drift against the rungs* — honest, and spec 007 open question 5's visualization-side answer. Mid-playback anchor switches (a mute toggle) snap the grid — coherent, since the audible stream changed.
> **Dissent:** *Product* wanted a user-pickable anchor; deferred — zero new UI in v1, and the deterministic rule is explainable ("the grid follows your most rhythmically confident audible track"). *Adversary* — an anchor whose `startTime > 0` (overdub) must have its ticks offset by it, and a deleted anchor must fall back cleanly; both get named tests.

> **Decision 4 — the arrival pulse is data-driven: persisted anchor ticks + `getEngineTime()`, superseding #485's live spectral-flux mechanism for playback.** A pure-state `BeatPulse` class (envelope: instant attack on tick crossing, exponential decay, same ballistics family as `BarSmoother`) is advanced in `useScrubberScroll`'s existing rAF loop and rendered as the meter frame's glow/width by `loudnessMeterRenderer`. It resets wherever `renderLoudnessMeterIdle` fires (pause/stop/seek — the #483 lesson applied at design time, not discovered in review). Beats only — onset-driven frame pulsing is dropped: onsets are already visible as ticks crossing the line and (in production) as the bars themselves jumping.
> **Rationale:** *Verification decides.* #485's detector consumes live CQT frames that this sandbox never delivers (#542) — its envelope would be unit-testable but its rendering unfalsifiable end-to-end here. The data-driven pulse is deterministic (fixture ticks + engine time), verifiable at unit level (envelope transition traces) *and* e2e level (poll engine time past a known beat, decode meter-frame pixels — the #484 ritual), and has *better* timing than rAF-sampled flux (no detection jitter). It also works when the categorical thing it renders — the beat — is what the listener taps, not every transient.
> **Dissent:** *Adversary* — the live path's death means no pulse during live recording input and no pulse on tracks whose analysis hasn't completed; both accepted (recording visuals were already out of scope in spec 003; pre-analysis is a transient state that degrades to today's rendering). #485 is re-scoped by a comment, not closed silently: its pulse mechanism is superseded here; spec 003's QA-handoff tasks in it stay with spec 003.

> **Decision 5 — anticipation renders detected ticks plainly (the strip already previews content) and extrapolates only where detection is silent — ghosted, bounded, and last.** Detected rungs render in the anticipation strip exactly like tiles do — the strip's perceptual slightness (perspective-compressed toward the horizon, the minority of the runway's visual weight) is the built-in honesty budget. Phantom rungs (Goal 5): extrapolate from the median of the last k detected inter-beat intervals, only beyond the last detected tick or across a detected gap, only while the anchor holds confidence, at a bounded horizon (a few intervals), at distinctly lower alpha. Shipped as the final cuttable milestone.
> **Rationale:** a "pure induction" strip (hiding detected future ticks, showing only extrapolation) was rejected — rungs vanishing above the line while spectrogram tiles continue through it would read as a rendering bug, not a philosophy. Phantom continuation is the one place extrapolation is *more* honest than silence: meter is retrospectively induced and felt as continuing (the owner's frame), so a ghost rung where the listener's expectation strikes renders their own percept — but it's decoration if QA doesn't feel it, hence cuttable and last.
> **Dissent:** *Simplicity* — cut Goal 5 entirely if any milestone runs long; nothing downstream depends on it. *Adversary* — extrapolation on rubato produces confident-looking nonsense; the confidence gate plus horizon bound contain it, and the arrhythmic fixture's absence test proves the gate.

### Architecture notes

- **`features/rhythm/`** (new feature directory, or unified with #559's tempo module if that lands first — one worker, one essentia load): `RhythmAnalyser` (worker round-trip), `selectRhythmAnchor`, `BeatPulse`, `extrapolateTicks` — all pure/deterministic except the worker call. Analysis triggers where melody extraction triggers (upload complete, restore-when-absent), with the #540 existence guard on write-back.
- **Persistence:** `rhythms` store `{trackId, ticks, onsets}` (+ scalars stay on the track record per #559); `DB_VERSION` 4; store name added to `TRACK_DATA_STORES` only — no standalone delete calls.
- **Rendering:** `RhythmOverlay.tsx` + `rhythmOverlayRenderer.ts` (rungs + phantom rungs; pure draw functions of `(ticks, anchorStartTime, scrollTop, pps, window)`); onset ticks in `Spectrogram.tsx`'s overlay write path (`writeRhythmTicks` beside `writeMelodyOverlay`). All marks apply `Track.startTime`. Zoom LOD: pure function `visibleRungStride(pps, medianIBI) → 1|2|4`.
- **Bridge:** `window.__mawimbi` gains rhythm read access (via `spectrogramCache.getRhythm` — no new top-level key needed if the cache carries it).
- **No new UI controls.** Rendering is data-driven and appears when analysis lands, like the piano roll.

## Open questions

1. **Rung layer order** — above or below track content (floor markings vs overlay)? A CSS/renderer constant; resolved by human QA on device, not in code review.
2. **SuperFluxExtractor vs OnsetRate** — M1's shape validation picks by output quality on the click and swung fixtures; recorded in `kb/decisions.md`.
3. **Does the anchor need surfacing?** V1 renders with zero UI; if QA finds the grid's provenance confusing ("why is the grid following *that* track?"), a subtle affordance (e.g. a badge in the effects drawer next to #560's BPM display) is the follow-up — evidence: QA checklist.
4. **Default densities and alphas** — LOD thresholds, tick length/alpha, rung alpha, ghost-rung alpha are named constants; QA tunes on device (the `?tune` overlay pattern, #447, is available if tuning by constants edit proves too slow).
5. **Anchor churn during mute-toggling** — the grid snapping on anchor change is chosen behavior; if QA finds it jarring, the fix is a short cross-fade in the renderer, not a different anchor rule.
6. **Should onset ticks render for muted tracks at all?** They currently inherit the muted track's hidden/dimmed styling for free; if QA wants muted tracks' rhythm fully invisible, it's a draw-path filter — decide with evidence, not now.
