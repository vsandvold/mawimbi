# 006 — Spectrogram rendering: performance, memory, and live analysis foundation

**Status:** In progress
**Date:** 2026-07-22
**Issues:** #537 (tracking) → #538 (harness), #539 (progressive analysis), #540 (cache lifecycle), #541 (render loop + fade), #542 (live analysis), #543 (effects preview), #544 (close-out)

## Summary

Makes the spectrogram subsystem — the product's visual identity (`kb/product.md`) — scale to long tracks (3+ minutes) and multi-track sessions while staying interactive: tiles appear progressively instead of after full-track analysis, memory stops growing without bound, one shared render loop replaces N always-on per-track loops, live frequency analysis consolidates onto the audio thread (one infrastructure serving the playhead meter, the future onset detector, and the recording preview), and dragging an effects slider previews the actual post-effect spectrogram in the visible window without waiting for a commit. This is the foundation layer for the engaging, interactive frequency visualizations the owner wants to keep building.

## Grounding

Current implementation (verified in code this session):

- **Offline path:** upload → `useSpectrogramCache` → `SpectrogramCache.analyse` → worker (`spectrogram.worker.ts`: CQT + tile rendering + Basic Pitch melody) → `TrackSpectrogramEntry` → IndexedDB `spectrograms` store → `Spectrogram.tsx` rAF draw. Analysis constants: 40 frames/s (25 ms hop), ~225 bins at 44.1 kHz (`CQTAnalyser.ts`). A tile is a transposed `ImageBitmap`, width = bin count (~225 px), height = `TILE_FRAMES` = 4096 frames = **102.4 s of audio per tile** (`Spectrogram.tsx:22`, `SpectrogramTileRenderer.ts`) — ~3.5 MB GPU each. **No tile is available until the whole track is analysed**; time-to-first-visual scales with track length.
- **Rendering:** one unconditional 60 fps rAF **per mounted track** (`useAnimationFrame.ts` — no gate), each frame doing a `getComputedStyle` + DOM query in `getCanvasWindow` (`Spectrogram.tsx:197–226`) even when idle. Visible-window culling already exists: the canvas bitmap is window-sized and only intersecting tiles are blitted (`drawTilesFrame`, `Spectrogram.tsx:352–420`); the dirty check compares the tiles array by reference (CLAUDE.md gotcha, #494). The melody overlay is a second window-sized canvas that redraws every frame during playback.
- **Memory:** each cache entry retains **both** the raw per-frame `Uint8Array[]` columns (~540 KB + 2400 array objects per audio-minute) **and** the GPU tiles (~2 MB/min) for the whole session (`SpectrogramCache.ts:81`). `invalidate`/`invalidateAll` exist but are **never called from production code** (grep-verified) — deleting a track or navigating between projects evicts nothing; tiles from previous projects leak until reload. The IndexedDB `spectrograms` store has no eviction or quota check.
- **Live/recording path:** recording analyses the mic on the **audio thread** (`AnalysisProcessor.accumulateAndAnalyseCQT`, transferable frames) into a `RecordingBuffer` `OffscreenCanvas`; `drawRecordingFrame` writes `style.height`/`marginBottom` then reads `offsetTop`/`getComputedStyle` in the same rAF tick — one forced synchronous layout per frame per visible track (#469 item 2). On stop, live rendering is discarded and the track re-analyses offline (live-then-refine, `kb/domain.md`).
- **Playback meter path:** `useScrubberScroll.ts:133` constructs `FrequencyVisualizer` **without** the worklet analyser, so during ordinary playback a full ~225-bin CQT of the master bus runs **on the main thread every frame** (`LiveCQTAnalyser` via `AnalyserNode` fallback) — redundant with the audio-thread CQT infrastructure recording already uses. The meter consumes these frames via `poolSemitoneBars`/`barTransfer` (spec 003); the planned `OnsetDetector` (#485) will consume the same stream.
- **Effects refresh (spec 004 M6, #494):** commit-only, 400 ms debounce, then **full-track** `renderTrackOffline` + full CQT + full re-tile + full IndexedDB rewrite (`effectsRefresh.ts`) — cost independent of the change's size; nothing at all happens during the drag. Supersede logic (monotonic request id) already prevents stale overwrites. Spec 004 explicitly chose honest visuals (tiles show actual post-effect audio) with commit-debounce to avoid per-tick re-analysis.
- **Zoom** (50–800 px/s) is draw-time `drawImage` scaling — cheap, no re-render, no observed artifacts.
- **Dead code:** `SpectrogramCanvasRenderer.ts` and `spectrogramRenderer.ts` are imported nowhere; the FFT analysis mode (`fft.ts`, `AnalysisProcessor`'s `frequencyEnabled` path, `WorkletAnalyser.enableFrequencyAnalysis`) is exercised only by tests — production is CQT-only. Verified: `fft.ts` is not used by the CQT path (kernels arrive pre-serialized).

## Goals

1. **Progressive tiling.** Tiles become visible incrementally during analysis: on a 3-minute upload, the first tile (covering the start of the track, where the playhead sits) is in the cache entry before full-track analysis completes, and chunked analysis output is byte-identical to the current single-pass output.
2. **Bounded memory with observable accounting.** After a track's spectrogram is persisted, its raw `frequencyFrames` are released from the in-memory entry (tiles remain); deleting a track or leaving a project closes its tiles and removes its entries. A DEV bridge exposes per-track and total tile/frame byte counts so the bounds are measurable, not asserted.
3. **One shared render loop.** All spectrogram canvases (including a recording track's) draw from a single rAF loop that reads the canvas window once per frame; frames where nothing changed (no scroll/zoom/tiles/playback/recording activity) perform no DOM reads or draws.
4. **Recording draw without forced layout.** The recording-frame path batches all DOM reads before all DOM writes within the shared loop's frame (#469 item 2), preserving today's visual behavior (live spectrogram grows with the recording; live-then-refine on stop unchanged).
5. **Live analysis on the audio thread.** During playback, the playhead meter's CQT frames come from the audio-thread worklet (same `AnalysisProcessor` infrastructure recording uses, tapped at the destination); the main-thread `LiveCQTAnalyser` remains only as the fallback when worklet init fails. The frame stream is exposed so the meter, the future `OnsetDetector` (#485), and spec 005's live pitch can all consume it.
6. **Live effects preview while dragging.** While a slider in the effects drawer is being dragged, the visible window of the active track's spectrogram updates (throttled, latest-wins) with an honest windowed post-effect render — actual `renderTrackOffline` + CQT of a capped window around the visible region, drawn as a provisional overlay. The preview never enters `SpectrogramCache`, never persists, and never carries an effects hash; the commit path (spec 004 M6) is unchanged.
7. **Far-edge fade.** Spectrogram content fades over its last pixels toward the runway's far edge instead of cutting hard (#468 option 2), on tracks longer than the runway window.
8. **Dead code removed.** `SpectrogramCanvasRenderer.ts`, `spectrogramRenderer.ts`, `fft.ts`, and the FFT analysis mode in `AnalysisProcessor`/`WorkletAnalyser` are deleted; the suite stays green.

## Non-goals

- **WebGL tile compositing (#163).** Canvas 2D blitting is not a measured bottleneck; M1's draw-time counters make it measurable. Trigger: revisit if blit time for 5 tracks exceeds ~4 ms/frame on a mid-range phone.
- **Zoom mipmapping (#160).** Zoom is draw-time scaling with no observed artifacts across 50–800 px/s. Trigger: visible artifacts on device at zoom extremes (QA checklist item).
- **Tile prefetching (#162).** Obsolete under the all-tiles-up-front design — propose closing.
- **Melody overlay toggle (#301), live pitch (#302/#527/#528).** Separate features; spec 005 owns live pitch. This spec only ensures the consolidated live stream can host it.
- **Extending track visuals beyond original duration for effect tails.** Spec 004's non-goal stands; the preview renders at original duration too.
- **Beat tracking (#165)** and any new visualization features — this spec is the foundation, not new spectacle.
- **Changing the CQT itself** — bins, hop, kernel caps are settled (`kb/domain.md`, #239/#242); every new path must keep passing the 12-TET guard.

## Design

### Decision 1 — Live effects preview: honest windowed render into a provisional overlay, never into the cache

> **Decision:** A `PreviewScheduler` (sibling of `EffectsRefreshScheduler`) runs while a slider drag is live: throttled (~150 ms, latest-wins), it calls `renderTrackOffline` on a **capped window** of the track's audio (visible canvas window intersected with the track, capped at `PREVIEW_WINDOW_MAX_SECONDS` ≈ 12 s, plus `PREVIEW_PREROLL_SECONDS` ≈ 2 s of lead-in to warm reverb/delay state, discarded from the output), runs CQT + tile rendering on just that slice in the existing worker, and hands the resulting bitmap to the draw loop as a **provisional overlay** drawn above the dry tiles for exactly that time range, edges feathered. On commit, the existing debounced full refresh runs unchanged and the overlay is cleared when it lands; on cancel/undo, the overlay is simply dropped.
> **Rationale:** honest visuals was spec 004's settled decision — the preview shows real post-effect audio, just windowed. Keeping the preview **out of `SpectrogramCache`** eliminates by construction the #494 bug class (a provisional render can never be persisted or stamped with a hash claiming it matches committed effects), and means no changes to entry/hash semantics at all. Cost is bounded by the window, not the track: ~14 s of offline render + ~560 CQT frames per preview tick, independent of track length. The playhead meter already previews live effects for free during playback (it taps the destination) — the overlay complements it for the stopped case, which is how users actually tune a sound (spec 004's edit mode works while stopped).
> **Dissent:** *Simplicity* argued for no tile preview at all (meter-only, status quo) — rejected because the request is explicit and the meter shows nothing while stopped. *Adversary* flagged: reverb IR regeneration is async and `Math.random`-seeded on every preview render (e2e must not pin `Math.random`, `kb/verification.md`); zoom/scroll mid-drag changes the window — the preview targets the window captured at each tick, it does not chase scroll; a preview tick racing the commit refresh must lose (overlay cleared on refresh completion, checked by request id). *Performance* predicted the windowed render may still exceed the throttle on low-end phones at minimum zoom — accepted: the window cap bounds it, constants are named and tunable, and the QA checklist measures it on device before tuning (open question 1).

### Decision 2 — Live analysis consolidates on the audio-thread worklet; main-thread CQT becomes fallback-only

> **Decision:** The playback path stops constructing a bare `FrequencyVisualizer(destination)`: `AudioService` (or a small `LiveAnalysisService` it owns) provides a destination-tapped `WorkletAnalyser` using the same `AnalysisProcessor` class recording uses (native-context chain per the CLAUDE.md Tone-context gotcha), and `useScrubberScroll` consumes its frames. The recording path keeps its own mic-tapped instance. Both share one processor class, one message protocol, one CQT kernel serialization. `LiveCQTAnalyser` (main-thread) survives only as the automatic fallback when worklet init fails — the same silent-fallback shape `WorkletRecorder` already has. The frame stream is a service-level accessor so `loudnessMeterRenderer`, the future `OnsetDetector` (#485), and spec 005's pitch path all read the same frames.
> **Rationale:** the main-thread ~225-bin CQT per frame during playback is the single largest live main-thread cost (survey, this session), and the audio-thread path already exists, shipped, and is proven during recording — this is consolidation, not new machinery. It also gives #485 and spec 005 one place to plug in instead of three.
> **Dissent:** *Simplicity* proposed reading **cached offline frames** at playhead time instead of any live analysis (zero live DSP) — rejected: it breaks the meter during recording/monitoring (no offline frames exist yet), and it abandons the meter's "taps the destination, therefore always honest" property that spec 004 leaned on. Recorded as the fallback direction if *Performance*'s risk materializes: running destination CQT (playback) and mic CQT (recording) simultaneously doubles audio-thread analysis load during overdubs — if on-device QA shows dropouts, the playback tap pauses during active recording (the meter can show the mic analysis instead, which is what the user is performing) rather than reverting to main-thread analysis.

### Decision 3 — One timeline render loop with a registry, not N gated per-track loops

> **Decision:** A single `TimelineRenderLoop` (module in `features/spectrogram/` or `workstation/`) owns the one rAF. Each `Spectrogram` registers a draw callback on mount and unregisters on unmount. Per frame the loop: (1) computes the canvas window **once**; (2) runs a cheap global change check (phantom `scrollTop`, `pixelsPerSecond`, per-track tiles identity, playing/recording flags — all snapshot reads); (3) if nothing changed, returns without touching the DOM further; (4) otherwise runs registered callbacks in two phases — **measure** (all DOM reads) then **draw/write** — which is also what removes `drawRecordingFrame`'s per-frame forced layout (Goal 4). Per-track dirty checks (tiles by reference, offset, pps — CLAUDE.md) stay inside the callbacks.
> **Rationale:** the scene is conceptually one draw (the runway) — N independent loops each re-deriving the same window is structural duplication, and the idle cost (N × `getComputedStyle` at 60 fps forever) is pure waste on mobile. The registry is small (~50 lines) and the imperative-positioning model this builds on is already the settled design (#464).
> **Dissent:** *Adversary* flagged registry lifecycle (a leaked callback after unmount draws on a dead canvas) — covered by a unit test on register/unregister and the existing e2e suite; and warned the global "nothing changed" check must include recording state or the live preview freezes — recording tracks bypass the idle short-circuit. *Simplicity* preferred just gating each existing per-track loop — rejected because it keeps N window reads per frame and leaves the measure/write interleaving (Goal 4) unsolved per-track.

### Decision 4 — Progressive analysis via chunked emission; tile height drops 4096 → 1024 frames

> **Decision:** The worker analyses in chunks and posts tiles incrementally (first chunk first — the track's start is what's visible at upload). `TILE_FRAMES` drops from 4096 (102.4 s) to 1024 (25.6 s), so the first visual lands after ~1/8 the analysis of a 3-minute track and partial results are useful. Chunk boundaries overlap by the kernel's reach (`MAX_KERNEL_HOPS`) so chunked output is **byte-identical** to single-pass output — guarded by a unit test, plus the standard 12-TET spacing test for any touched mapping. Each incremental delivery goes through `SpectrogramCache.setEntry` with a **fresh tiles array** (the #494 reference-identity contract). Raw frames still accumulate during analysis (persistence needs the full set once) and are released per Goal 2 after the save.
> **Rationale:** time-to-first-visual currently scales with track length — the one structural blocker for "support 3+ minute tracks feeling instant". Chunking the existing pipeline is strictly smaller than any streaming redesign; smaller tiles also shrink the preview overlay's blast radius and cost nothing measurable in GPU overhead (same total bytes, 8× the bitmap count).
> **Dissent:** *Adversary*: chunk-seam bugs are the classic failure — hence byte-identity as the acceptance test, not "looks right". *Simplicity* asked whether progressive melody extraction should follow — no: melody is an overlay nicety with its own worker path; out of scope.

## Open questions

1. **Preview constants** (window cap, preroll, throttle): are 12 s / 2 s / 150 ms right on a mid-range phone? Resolved by M1 baseline + M7 QA measurement; constants are named and tunable without touching tested logic.
2. **Recording-stop continuity:** after M2, is the progressive first tile fast enough that the post-recording blank gap doesn't register, or should live frames seed a provisional tile at stop? Evidence: M2's `firstTileMs` for a 3-min recording on device. Only build the seeding if the gap is visible.
3. **Overdub audio-thread headroom** (Decision 2 dissent): does destination CQT + mic CQT simultaneously cause dropouts on low-end devices? Evidence: M7 QA. Fallback plan recorded in Decision 2 (pause the destination tap during active recording).
4. **IndexedDB quota policy:** M3's audit will show how much orphaned spectrogram data real projects accumulate; decide then whether `StorageUsage` needs a spectrogram-specific surface or an LRU. Don't build speculative eviction.
5. **Preview seam treatment:** does the feathered boundary between previewed and dry regions read as intentional? M7 QA; fallback is extending the preview window to the full canvas window on capable devices.
6. **`imageSmoothingEnabled` when stretching tiles at high zoom** — currently bilinear by default; is crisper nearest-neighbor better at 800 px/s? M7 QA checklist item (relates to #160's trigger).
