# 005 — Recording: input monitoring, live pitch, punch-in/out

**Status:** Draft
**Date:** 2026-07-21
**Issues:** (filled by /spec-to-issues)

## Summary

Extends the recording feature along the three lines the open issue backlog plans for it: a dedicated recording drawer that becomes the feature's home in the workstation UI, input monitoring while recording (#171), punch-in/out recording over an existing track (#174), and — as an explicit stretch phase — live pitch display on the in-progress recording (#302, reconciled with the Basic Pitch migration; two candidate designs recorded, choice deferred). The spec also documents the current recording solution as the baseline it builds on — including the live spectrogram during recording, which already shipped and needs no new work. Target user is the creative amateur overdubbing takes on a phone or laptop (`kb/product.md`).

## Current solution (baseline)

What exists today, so requirements below are diffs against reality rather than re-plans of shipped work:

- **Workflow** — GarageBand-style `idle → armed → recording → idle` state machine (`RecordingService.ts`); the deliberate arm → position → play sequence is a product rule, not an accident (`kb/product.md`). Arm via toolbar, then a 4-beat visual count-in (~120 BPM) masks mic-permission and recorder startup latency while playing existing tracks as context (#199; `useCountIn` in `workstationEffects.ts`). The transport is locked and spacebar blocked during count-in/recording.
- **Capture** — AudioWorklet PCM capture (`WorkletRecorder` + `RecordingProcessor`) on the extracted native `AudioContext`, with `Tone.Recorder` as silent fallback (#209, #210; decision 2026-03-03 in `kb/decisions.md`). Low-latency `getUserMedia` constraints bypass `Tone.UserMedia.open()` (#219; `MicrophoneService.LOW_LATENCY_CONSTRAINTS`). Mono, echo-cancellation/noise-suppression/AGC off.
- **Alignment** — `LatencyCompensation` trims the recorded buffer by `outputLatency + baseLatency + lookAhead + one render quantum` (#209); trimming the buffer rather than passing an offset downstream is deliberate (#212, `kb/domain.md`).
- **Live spectrogram** — already shipped. `FrequencyVisualizer` (worklet CQT preferred, main-thread CQT fallback) feeds `RecordingBuffer`, which accumulates one pixel row per ~25 ms hop on the recording track's canvas; when recording stops, the rough live rendering is replaced by the polished offline tiles — the live-then-refine pattern (`kb/domain.md`, "Two analysis paths"). This spec adds no live-spectrogram work; the live pitch display (Goal 3) draws on the same recording canvas but gets its data from a different source (Decision 2), not from this CQT pipeline.
- **Completion** — `stopOverdubRecording()` returns the trimmed buffer + transport start time; `useMicrophone` creates a new track (`createRecordedTrack`), persists audio, dispatches `ADD_TRACK`, and pauses at position. Melody transcription (Basic Pitch) then runs in the background like for any track (`useSpectrogramCache.ts`), so the piano-roll overlay appears on recorded tracks after a delay — the "post-recording refinement" half of #302 effectively already ships.
- **Not routed to output** — the microphone connects only to its meter/analyser, never to `Tone.Destination`; there is currently no software feedback loop (#171's own analysis).

## Goals

1. **Deterministic capture proof:** an e2e test records from a synthetic 440 Hz microphone stream and proves, via the `window.__mawimbi` bridge, that the resulting track transcribes to MIDI 69 — closing the loop capture → latency trim → track creation → transcription.
2. **Input monitoring (#171):** while the microphone is open (count-in and recording), the user can toggle hearing themselves through the output, with a monitor volume slider. Off by default; enabling shows a feedback warning; a measured round-trip latency > 50 ms shows a latency warning.
3. **Live pitch display (#302, live half) — stretch:** during recording, a pitch visualization renders on the in-progress recording's canvas — either incremental Basic Pitch notes or a frame-rate pitch trace, per the deferred Decision 2. Whichever design ships must not fall progressively further behind real time on a device where analysis is slower than capture.
4. **Refinement continuity (#302, refine half) — stretch:** when recording stops, the live pitch visualization is discarded with the live spectrogram and the existing full-buffer Basic Pitch pass takes over — no MELODIA work.
5. **Punch-in/out (#174):** the user can re-record a section of an existing track: pick the target track in the recording drawer, position the playhead, arm punch, count in, record, stop — and the recorded span is spliced into that track's audio (crossfaded at the boundaries, latency-compensated), replacing the original content in that range. The splice is undoable and restores the original audio exactly.
6. **No behavioral regression:** the recording lifecycle (arm, count-in, record to a new track, cancel, transport lock) keeps its current behavior — only its entry point moves (Goal 7); the recording/audio e2e suites stay green (updated for the new entry point where they drive the old one).
7. **Recording drawer:** the recording feature lives in a new bottom-sheet drawer alongside the mixer/lyrics/effects sheets — opened from the toolbar mic button, hosting the record/stop control, mic level meter, count-in state, the Goal 2 monitoring controls, and (from Milestone 5) the Goal 5 punch controls. The drawer stays open through count-in and recording; other sheet toggles stay locked out while the transport is locked, exactly as today.

## Non-goals

- **Multi-take recording (#175)** — needs a take-stack data model; separate spec when prioritized.
- **Timeline range markers for punch** — #174 as written asks for in/out markers on the timeline; deferred. Draggable markers on the tilted runway conflict with the drag-only-seek decision (`kb/decisions.md` 2026-07-18) and the known unreliability of hit-testing on the 3D plane (`kb/verification.md`), and useful markers want beat-grid snapping (#165). Punch range is transport-anchored instead (see Design). Revisit markers when #165 lands.
- **Manual latency calibration** — the click-track calibration tool sketched in #170 was deliberately not built when the computed compensation shipped (`kb/domain.md`); that stays the mechanism.
- **Tempo-aware count-in** — awaits beat tracking (#165); count-in stays 4 beats at ~120 BPM.
- **essentia `PitchYinFFT` in the AudioWorklet** — #302's implementation sketch predates the Basic Pitch migration and would introduce a WASM load path into the audio thread; both candidate designs in Decision 2 avoid it. #302's MELODIA refinement step is likewise superseded — refinement is the shipped Basic Pitch path.
- **Stereo capture, monitoring effects (reverb-in-headphones), recording-time metronome click** — out of scope.

## Design

### Decision 1 — Punch model: transport-anchored destructive splice, no marker UI

> **Decision:** Punch is an action on a specific existing track, armed from the recording drawer (Decision 5; owner direction 2026-07-21): the drawer's punch control includes a target-track selector and is enabled only when the playhead is at or after the selected track's start. The user positions via the existing scrubber and the normal count-in flow runs. Punch-in = the engine time when capture starts; punch-out = when the user stops (extension past the original track end is allowed and grows the buffer). The recorded span is spliced destructively into the track's `AudioBuffer` with short equal-power crossfades at the boundaries, via a pure `spliceBuffer()` function and a new `TrackService.replaceTrackAudio()` + `REPLACE_TRACK_AUDIO` reducer action whose reverse action restores the previous audio.
> **Rationale:** Architect: this reuses the arm → position → count-in → record machinery wholesale and mirrors the shipped undo pattern (command history with reverse actions; `AudioSourceRepository` buffers are deliberately retained in-session, so holding the pre-punch buffer for undo is free — `kb/decisions.md` 2026-02-22). Passing a whole replacement buffer, not an offset, follows the #212 lesson. Simplicity: no new data model (a track stays one buffer + startTime) and no new timeline interaction surface. Product: "fix this bit, starting here" matches the amateur's mental model; range markers are pro-DAW furniture. Housing punch in the recording drawer keeps every capture flow in one place and avoids an edit-mode → recording handoff entirely — spec 004 already forbids those states overlapping, so arming punch from the edit sheet would have required a sheet transition anyway. A non-destructive region/overlay model was rejected: it changes the track data model project-wide for a feature undo already covers.
> **Dissent:** Product (minority) argued visible in/out markers communicate the punch range better than an invisible transport anchor, and that stop-as-punch-out risks sloppy punch tails. Recorded as the trigger to revisit when #165's beat grid enables snapping markers. Adversary flagged the splice-position math as the likely bug site: punch positions are engine-time based (`playback.getEngineTime()`, never `transportTime` — the CLAUDE.md gotcha) and must be converted to buffer-local offsets against the track's own `startTime` after latency trimming; this is why the splice is a pure function with sample-accurate unit tests before any UI exists.

### Decision 2 — Live pitch: stretch phase; two candidate designs kept, choice deferred

> **Decision:** Live pitch is deliberately sequenced as a **stretch phase after input monitoring and punch ship** (owner call, 2026-07-21). The reasoning: with the post-stop overlay already appearing seconds after recording ends, live pitch's incremental value is during-take reassurance/delight — real, and on-brand for "visualize sounds like you hear them", but not a workflow gap like "I can't hear myself" (#171) or "I must redo the whole take" (#174). Two implementations were designed and **both are kept as live alternatives**; the choice is made at the start of the stretch phase, not now:
>
> **(A) Incremental Basic Pitch** — reuse `@spotify/basic-pitch` (already wrapped by `MelodyExtractor`, transcription worker, decision 2026-03-09). Tee the PCM chunks `WorkletRecorder` already posts to the main thread into the transcription worker; run `evaluateModel` per inference window (~2 s, matching the lib's own internal windowing: `AUDIO_WINDOW_LENGTH_SECONDS = 2`, 30-frame overlap, hop ≈ 1.65 s in `inference.ts`), posting back note events with the same thresholds and `MelodyNote` shape as offline extraction; skip-to-latest scheduling bounds lag at ~one window. Pros: the live display shows *the same polyphonic notes* the final overlay will show (no rough-vs-refined mismatch), zero new DSP code, no audio-thread changes, slow inference degrades to display lag rather than audio dropouts. Cons: notes lag ~1.6–2 s (cannot serve real-time intonation feedback), TF.js inference on a phone during capture is unmeasured, window-boundary segmentation differs from the final pass until stop.
>
> **(B) Pure-TS YIN trace in `AnalysisProcessor`** — a cumulative-mean-normalized-difference YIN detector (~150 lines) alongside the existing pure-TS CQT in the worklet, enabled only while recording, posting `{ type: 'pitchData', frequency, confidence }` at the ~25 ms CQT hop cadence; drawn as a thin trace next to the live spectrogram. Pros: near-real-time — the only design that can serve karaoke-style intonation feedback while singing; monophonic fundamental estimation by construction (unlike CQT-argmax, which picks the strongest harmonic and is capped below ~340 Hz, #242); unit-testable as a pure function. Cons: a second pitch algorithm to maintain, a monophonic rough trace that will visibly disagree with the polyphonic refined overlay, adds work to the audio thread.
>
> **Rationale for deferring:** the two designs serve *different jobs* — (A) delight/consistency, (B) intonation coaching — and which job matters is a product judgment better made with a shipped monitoring + punch build in hand. Deferral costs nothing: both designs are fully specified here, neither blocks the other milestones, and the Milestone 1 harness (440 Hz fake mic + bridge reads) verifies either.
> **Resolution criteria:** (1) the owner picks the job — reassurance/delight → A; intonation feedback → B; (2) if A, the throughput measurement issue (Verification, Goal 3) must show a ~2 s window inferring in < ~1.65 s on a mid-range phone during capture, else fall back to B; (3) if neither job seems worth two milestones once monitoring + punch are in hand, close the stretch phase and move Goals 3–4 to a non-goal pointing at #302.
> **Dissent:** rejected outright either way: essentia `PitchYinFFT` in the AudioWorklet (#302's original sketch — unproven WASM load path on the audio thread) and CQT-argmax (systematic octave errors on harmonic-rich sources). Adversary notes the deferral risk is re-litigating this choice from scratch later — mitigated by this record: a future session re-opens the decision with these criteria, not a blank page.

### Decision 3 — Input monitoring: a monitor gain inside MicrophoneService, warn-don't-block

> **Decision:** `MicrophoneService` owns a `monitorGain` (`Tone.Gain`) that, when monitoring is enabled, connects the existing `Tone.UserMedia` source to `Tone.Destination` — entirely inside the Tone graph, independent of the native-context capture path. A headphone-icon toggle plus volume slider live in the recording drawer (Decision 5), active whenever the mic is open (so monitoring works during count-in, where the mic already opens early — `kb/product.md`). Monitoring defaults to off each session. Enabling it shows a feedback warning toast (headphone detection on the web is unreliable, so warn always rather than gate); if `estimateRoundTripLatency()` exceeds 50 ms, a latency warning shows too (#171's thresholds).
> **Rationale:** Architect: the mic already lives in the Tone graph (its meter connection proves the wrapper path works); only the recording capture path needs native nodes, and monitoring never touches it. Adversary: echo cancellation is deliberately off for music capture (#219), so speaker feedback is a real and fast failure — hence default-off and an unconditional warning; a "detected headphones" gate would be false confidence built on `enumerateDevices` label heuristics. Simplicity: no attempt at zero-latency native monitoring paths until the measured Tone-graph latency is shown to be a problem on real devices (that measurement is the human-QA issue).
> **Dissent:** Performance argued for building the monitor chain on the native context (like the recorder) to shave wrapper overhead; rejected as speculative — no measurement shows the Tone path's latency is worse, and the native path costs the established-but-fragile `_nativeContext` extraction for no proven gain. Recorded so the on-device QA issue checks perceived monitoring latency explicitly.

### Decision 4 — Verification: replace the fake beep with a controllable synthetic microphone

> **Decision:** Milestone 1 builds `e2e/helpers/fakeMicStream.ts`: a `page.addInitScript` override of `navigator.mediaDevices.getUserMedia` that returns an `OscillatorNode → MediaStreamAudioDestinationNode` stream at a known frequency, with a `window.__setFakeMicFrequency(hz)` control for changing pitch mid-test. All new recording e2e assertions build on it plus the existing `window.__mawimbi.spectrogramCache` bridge (melody notes are already readable through it).
> **Rationale:** Chrome's `--use-fake-device-for-media-stream` beep has no contractually known frequency, so it can prove "a track was created" (as `recording.spec.ts` does today) but not "the right audio was captured". A known tone turns capture, latency trim, splice, and live pitch into data-level assertions: recorded track transcribes to MIDI 69 (#480 proved Basic Pitch e2e is deterministic on pure tones); a punch with the oscillator switched to 660 Hz yields a track whose melody reads 440-tone before the punch range and 660-tone inside it — verifying the splice by listening to the result, not by trusting buffer math twice.
> **Dissent:** Simplicity noted the override must be installed before app scripts run and adds one more layer of fakery to maintain; accepted because every alternative (asserting on real beep content, exposing capture buffers wholesale through the bridge) is more fragile or wider.

### Decision 5 — The recording feature moves into a dedicated bottom-sheet drawer

> **Decision:** A new `RecordingBottomSheet` joins the mixer/lyrics/effects sheets: `'recording'` is added to the `ActiveSheet` union in `Workstation.tsx`, the toolbar mic button becomes the drawer's open/close toggle (replacing the current direct arm-toggle in `FloatingToolbar.tsx`), and the drawer hosts the record/stop control, a mic level meter (`RecordingService.getLoudness()` already exists), count-in state, and the monitoring toggle + volume from Decision 3. The drawer stays open through count-in and active recording — the runway remains visible above it, its geometry already adjusting to open drawers (`runwayProjection`); stopping happens in the drawer. Sheet exclusivity is inherited from the `activeSheet` mechanism, and the existing transport lock (`isTransportLocked`) keeps other toggles inert mid-recording, unchanged. The recording *services* are untouched — this is a UI relocation, not a state-machine change (owner direction, 2026-07-21).
> **Rationale:** Product: a single toolbar toggle cannot host #171's monitoring UI, a level meter, or the punch controls (Decision 1); recording is a core creative act and deserves a first-class home like the mixer and effects — and bottom sheets are the app's established mobile-first surface (`kb/product.md`, decision 2026-03-08). Architect: the sheet slots into the existing single-effect enter/exit pattern keyed on `activeSheet` (decision 2026-07-20 — the pattern exists precisely because per-handler enter/exit calls leaked state), and spec 004's "edit mode and armed-recording never overlap" invariant becomes structural: they are now competing sheets in one union rather than two independently-guarded states. Adversary: the known traps are catalogued and apply — the toolbar toggle is `pointer-events: none` while its own sheet is open (close via the sheet's own control in e2e, CLAUDE.md), and a "hidden" toolbar remains keyboard-activatable (#488 class), so the record control's guards must live in state, not reachability.
> **Dissent:** Simplicity: the current one-tap mic toggle is the fastest path from impulse to recording, and the drawer adds a tap before arming — a real cost for the amateur's immediacy. Mitigations considered (auto-arm on drawer open; a prominent single record CTA as the drawer's primary element) are left to implementation, but the concern is recorded: if on-device QA finds the extra step kills the impulse, the answer is a faster path *into* the drawer, not moving controls back out of it.

## Open questions

1. **Punch target-track selector form** — punch lives in the recording drawer (owner decision, 2026-07-21), which therefore needs a compact track picker: candidates are color chip + instrument icon rows (mirroring the mixer's channel identity) or a horizontal chip strip. Resolve during Milestone 5 implementation; the workflow behind the selection is unaffected.
2. **Live pitch design choice (stretch)** — A vs B per Decision 2's resolution criteria: the owner picks the job (delight → A, intonation feedback → B) once monitoring + punch are shipped; A additionally gated by the throughput measurement. If A: one full lib window (~2 s) per run is the default cadence; a shorter partial-window cadence (~500 ms re-inference) is adopted only if the measurement shows headroom.
3. **Crossfade length** — default 10 ms equal-power; ear-check on device in the Milestone 3 QA issue (add a punch item to the same checklist when Milestone 5 lands).
4. **Monitoring persistence** — per-session off is specced; if QA shows users re-enable it every time with headphones, consider persisting the preference per project (would need a product sign-off, not assumed here).
5. **Live pitch visual treatment (stretch, if design A)** — live notes are the same shape as the final overlay; whether they need a "provisional" treatment (e.g. reduced opacity until the final pass confirms them) is a design call for on-device QA. Start identical (simplest); add the treatment only if the stop-time re-segmentation visibly "jumps" in a way QA finds jarring. (Design B's thin trace is self-evidently provisional and needs no such treatment.)
6. **Drawer fast path to record** — Decision 5's dissent: the drawer adds a tap before arming versus today's one-tap toggle. Candidates: auto-arm on drawer open, or a dominant record CTA as the drawer's primary element. Resolve in the Milestone 3 on-device QA pass; the drawer's structure is unaffected either way.
