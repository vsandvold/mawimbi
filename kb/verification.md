# Verification

Catalog of verification patterns that work in this repo — and the ones that look like they work but don't. Every spec's verification design starts by consulting this file (`/spec` step 3) and ends by extending it with anything new that was learned. Mechanics (commands, jsdom limits, e2e environment quirks) live in CLAUDE.md's Testing section; this file is about *choosing* the right verification for a claim.

## Choosing a verification

Work down this list; prefer the earliest level that can actually falsify the claim:

1. **Unit test (vitest)** — service state machines, reducers, pure math (e.g. `runwayConfig.test.ts` sweeps every preset across viewport heights for NaN/Infinity). Read service state via plain getters.
2. **E2E invariant (Playwright)** — anything involving real layout, paint, stacking, scrolling, or cross-feature workflows. Assert *invariants* (alignment within tolerance, element visibility, geometry resync after resize/zoom) rather than pixel snapshots where possible — invariants survive intentional restyling.
3. **Screenshot-decoded pixels** — when even DOM geometry lies. Rect-based assertions pass straight through clipping bugs (rects ignore clipping); decoding actual screenshot pixels at a location is the only automated check that catches a "renders nothing" failure. This class caught the #459 runway clipping bug after rect-based tests stayed green (PR #464).
4. **Visual snapshot (`e2e/__screenshots__/`)** — pins an *intentional look* (per-preset appearance). Costs maintenance on every restyle; use for looks, not logic.
5. **Human QA issue** — what automation cannot see (on-device feel, address-bar viewport behavior, design judgment) gets an explicit checklist issue (pattern: #467). Never leave it silently untested; the escape hatch is a tracked issue, not a shrug.
6. **Profiling issue** — reasoned performance concerns become an issue with a concrete measurement plan and a close-if-comfortable clause (pattern: #469). Don't optimize unmeasured costs; don't silently ignore them either.

## Rules of thumb

- **Bugs get a failing test first** — CLAUDE.md, Bug Fixes. The reproducing test is what defines "fixed".
- **A computed value is not a rendered value** — jsdom's limits are catalogued in CLAUDE.md, Testing; stacking/pseudo-element/paint claims need e2e.
- **Distrust green tests around transforms and clipping.** If a change touches scroll containers, CSS transforms, or z-index, ask what a rect-based test would miss (see #464's visibility invariant).
- **`getBoundingClientRect` lies under 3D transforms** — it returns the axis-aligned bounding box of the *projected trapezoid*, not the logical rectangle, and Playwright's `hover()`/`click()` element-center can project off-viewport at steep tilt. Interact with untransformed wrappers (phantom/viewport) and scan actual canvas buffers instead of doing rect math (PRs #393, #395). Distinct mechanism from "rects ignore clipping" (#464); both mean rect assertions lie near transforms.
- **Reactivity/state-management migrations gate on the e2e suite, not the unit suite.** After the signals migration, all 233 unit tests passed while 0 of 69 e2e tests did (#114): signal effects run outside React rendering, so the audio engine kept responding while the UI silently stopped re-rendering — mock-heavy unit tests cannot see it.
- **Mock the real API shape, not your assumption of it.** Five consecutive classification bugs shipped past a fully green suite because mocks mirrored the code's assumptions about third-party modules (EssentiaWASM export style #272, object-not-factory #273, per-frame arrays #281, ONNX feed input names #284, EffNet's two outputs #295). When wrapping an untyped third-party module, verify the mock against the actual runtime shape (scratch script or e2e smoke) first, and prefer reading dynamic properties (`session.inputNames`, output dims) over hardcoding.
- **Don't assert implementation details.** ~3,000 lines of tests asserting canvas API call counts, worker message formats, node wiring, and RAF/ResizeObserver mock internals were deliberately deleted (PR #365), keeping user scenarios, state-machine transitions, and bug-fix regressions. Several renderers have no unit tests *on purpose* — don't "restore coverage" for their internals.
- **Frequency-mapping changes get a 12-TET test:** any new analysis/visualization path must place the same tone within ±2 bins of existing paths and keep semitone spacing uniform (spacing coefficient of variation < 15%) — the pattern that ended the live/offline misalignment bug class (#220; see kb/domain.md).
- **A spacing/uniformity test must call the function under test, not re-derive its expected output from the same formula the implementation uses.** A first draft of #482's semitone-spacing test computed `midiNoteToBin(midi) / 2` directly instead of feeding synthetic bins through `poolSemitoneBars` and reading the pooled output — it would stay green through an off-by-one or wrong-stride regression in the pooling loop itself, since it never exercised that code. Caught by code review before merge; fixed by pooling synthetic per-note bin arrays and asserting on `indexOf()` in the pooled result.
- **State-flap bugs need transition traces, not visibility polls.** A playback stutter (state flipping every ~250 ms, "playing" for only 1–2 frames per cycle) made `expect(locator).toBeVisible()` false-pass or false-fail depending on where the poll landed — in the same session, a visibility-poll repro test passed while the bug was live. The falsifiable form: inject an rAF sampler via `page.evaluate` that records `{t, title}` *transitions* of the play/pause button, then assert on the transition count (stable playback ⇒ ≤1 transition per user action). Pattern established in spec 002's bug investigation (2026-07-19); delivered as `e2e/helpers/flapTracer.ts` (`tracePlaybackState`/`stopPlaybackTrace`, re-exported via `e2e/fixtures.ts`), exercised by `e2e/playback-toggle.spec.ts` (#473, PR #496). **Corollary (#473):** don't gate the trace's start on a visibility-poll assertion for the state the triggering action produces — during a live flap bug that assertion inherits the same false-fail risk (a real run timed out waiting for `getByTitle('Pause')` because its ~15–40 ms windows fell between polls). Instead seed the trace *before* the triggering action and assert the total transition count allows exactly one (the action's own expected transition), not zero.
- **Flakiness protocol:** re-run the failing spec in isolation before blaming the change; `audio.spec.ts` has flaked under full parallel runs since 2026-03 (PRs #416, #418; reconfirmed in #457's test plan) — a long-lived flake, not a recent one.
- **Verify the fix against a live reproduction when possible** — e.g. PR #465 validated its recovery script against a real `node_modules` wipe, not a simulation.
- **Real Basic Pitch transcription is viable in e2e — measured, not assumed (#480).** Model self-hosted (`public/basic-pitch-model/`), so `fixtures.ts`'s `blockModelRequests` (which blocks `/models/*.onnx`) doesn't touch it. Measured: ~6–11s per upload (cold model load on a run's first upload, ~6–8s steady-state after) including navigation/upload/analysis, and three consecutive runs against the same fixture (`test-tone-short.wav`, a 440 Hz pure tone) produced the identical note (MIDI 69, same start/end time to the millisecond) — deterministic enough to assert on directly. No dev-only injection seam needed; use real transcription for melody-path e2e tests (spec 003's Milestone 4 sparkle tests should reuse this rather than reintroducing the injection-seam fallback).
- **`Tone.Reverb`'s impulse response is generated asynchronously and silently — verified against real Tone 15.1.22 in a browser (#489).** `new Tone.Reverb(...)` kicks off `generate()`, which renders the IR through its *own* nested `OfflineContext` and resolves the `ready` promise when done; until then `ConvolverNode.buffer` is `null`, which is silence, not an error or passthrough. Skipping `await reverb.ready` before starting the source inside a `Tone.Offline` render doesn't fail — it just quietly omits the reverb's contribution, so an assertion on "does the chain produce tail energy" can pass for the wrong reason if another chain member (e.g. `FeedbackDelay`'s echoes) supplies the tail instead. Isolate the effect under test (bypass the others) when asserting on one effect's contribution specifically. Also: `Tone.Filter` has no `wet` (it's a plain `ToneAudioNode`, not an `Effect` subclass) while `Tone.Reverb`/`Tone.FeedbackDelay` do — bypass-via-disconnect (spec 004 Decision 3) is required for `Filter` regardless, and is the uniform choice across all three.
- **Prefer a direct state read over screenshot-decoding when the claim is about data, not paint.** Screenshot-decoded pixels (level 3) exist for when paint itself is the thing being verified — data correctness (did extraction produce the right note?) is better proven by reading the state directly, which is level-1-equivalent even when done through a page-context bridge rather than a plain getter. `window.__mawimbi` (`src/global.d.ts`, wired in `AudioService.ts`, `import.meta.env.DEV`-gated) exists for exactly this: worker-produced state (melody notes) with no DOM/CSS surface a rect or pixel check could read. Reserve pixel-decoding for verifying that state actually rendered (e.g. the sparkle/pulse milestones).
- **A toggle button's e2e assertion races any concurrent async auto-transition — don't click "whatever the button currently says" as a timing-sensitive trigger.** An e2e test proving "an explicit command cancels a stale scheduled auto-resume" by clicking Play then Pause at the ~200ms debounce boundary was itself flaky: if the scheduled auto-resume fires in the same instant the click's handler runs, the toggle's *meaning* (start vs. stop) depends on which happened first, so the identical test intermittently passed or failed under load — a hazard of the toggle, not a bug in the fix it was testing (caught empirically: 4/4 failures after a passing run, traced with a flap-tracer debug spec). Prefer a control whose effect doesn't depend on current state (e.g. Rewind, which always forces stopped+0) to prove a cancellation mechanism deterministically regardless of timing (#475).

- **CDP's `Input.dispatchTouchEvent` can't reliably add a touch to an already-active sequence — reach for a jsdom unit test instead.** Reproducing "a second finger joins mid-gesture without the first lifting" (spec 002 milestone 4, #476) by dispatching a `touchStart` that lists the continuing point (matching its last position, even with an explicit `id`) alongside a new one does fire the native `touchstart` DOM event, but React's synthetic `pointermove`/`pointerdown` events stop firing entirely afterward in this environment — confirmed by instrumenting the real handlers. Don't spend more effort chasing CDP touch-point continuity for this shape of gesture; jsdom's `TouchEvent`/`Touch` constructors work fine for this (verified: `new TouchEvent` and `fireEvent.touchStart(el, { touches: [...] })` both work in this repo's vitest+jsdom setup), so drive the real (unmocked) component with `fireEvent.pointerDown/pointerMove` plus a native `fireEvent.touchStart`/`touchCancel` directly — it exercises the same production code path (`useTimelineZoom`'s real listeners) without depending on CDP's touch synthesis at all.
- **Not every geometry restyle moves `e2e/__screenshots__/`.** Spec 003's #481 (playhead meter: 0.4H→0.25H, centered→bottom-aligned, 2:1→3:1) changed only a thin translucent overlay well under `playwright.config.ts`'s `maxDiffPixelRatio: 0.01`, and the only full-page snapshot (`project-empty.png`) is taken before any track/playhead content exists — `visual.spec.ts` passed unchanged. Run the visual suite before assuming a snapshot update is owed; don't `--update-snapshots` reflexively.
- **Isolating one track's pixels in the "single mixed track" design: use time, not rects.** Every `.timeline__track` shares one CSS grid cell (`kb/domain.md`), so two tracks' wrapping divs have identical bounding rects — no rect-based clip can isolate one track's content from another's. Two tracks of the *same* frequency but *different* duration give a usable window instead: upload the longer one first (background), the shorter one second (active/newest), and sample the on-screen band for elapsed time inside `(shorterDuration, longerDuration)` — only the background track has content there. Force `page.emulateMedia({ reducedMotion: 'reduce' })` first so `scale(s) = 1` everywhere (`kb/domain.md`, Runway geometry) and elapsed time maps *linearly* to screen Y (`playheadLineY − elapsed × pixelsPerSecond`), avoiding the runway's real nonlinear projection. Established in spec 004 milestone 2's background-dimming assertion (`e2e/track-edit-mode.spec.ts`); the existing fixtures (`test-tone-short.wav`/`test-tone-long.wav`) are both 440 Hz, so this only works via duration, not frequency, until a differing-frequency short fixture exists.
- **`magnitudeToByte`'s dB range is `[-80, -30]`, not full-scale dBFS — a synthetic "loud" test magnitude needs a matching dB value or it silently clips.** A `barTransfer.test.ts` (#483) draft picked `magnitude = 1` for a "fundamental" and `10^(-10/20)` for a "harmonic 10 dB weaker", expecting a realistic ratio; both landed above `MAX_DECIBELS = -30` and clamped to byte 255, producing `NaN` (0/0) two steps later rather than a wrong-but-plausible number — the clipping was silent at the point it happened. Pick synthetic dB values from inside the analyser's actual range (e.g. `-35` and `-45`) when constructing CQT-byte test fixtures, and export the constants (`CQTAnalyser.ts` already exports `MIN_DECIBELS`/`MAX_DECIBELS`/`magnitudeToByte` for exactly this) instead of hand-rolling the byte math.
- **`git diff origin/master...HEAD -- <files>` can silently span more than "just the new commit" when a PR merge squashed history that a local branch kept unsquashed.** After PR #516 merged (squash or merge-commit unknown from the branch alone), continuing work on the same local branch left it with the pre-merge multi-commit history still intact, diverged in *shape* but not *content* from the now-squashed `origin/master`. A code-review pass scoped by `git diff origin/master...HEAD -- <changed files>` for "the new commit" actually included two already-merged, already-reviewed commits' content too — a finder agent caught this itself by comparing blob hashes between the intended single-commit diff and the actual reviewed diff. Before scoping a review diff on a branch that outlived a merge, confirm `git diff origin/<default> <branch-tip-before-new-work>` is empty (content-identical) rather than assuming the commit range matches commit *count*.
- **A toolbar toggle button is `pointer-events: none` while its own content sheet is open** (`ToolbarBottomSheet.tsx`'s `toolbar-sheet--hidden`, mirrored by every mixer/lyrics/effects e2e test that closes via `getByTitle('Close')` instead of re-clicking the toggle). Clicking the toggle's title text again to close a sheet hangs for the full test timeout with a `subtree intercepts pointer events` error, not a clear "button not found" — always close through the sheet's own Close button.
- **A translucent overlay's background is not a neutral backdrop — a plain saturation check can't tell its own accent color apart from bleed-through of whatever paints underneath it.** The playhead loudness meter's background renders at alpha 0.15 over the runway, so a track's own (randomized, #21/#36) spectrogram color shows through it. Building the sparkle-burst e2e test (#484), `hasSaturatedPixel` (the #464 pattern) flagged that bleed-through as a false positive for the new sparkle color even with `Math.random` pinned to a fixed track color — both are "saturated", so the check couldn't distinguish them. Fix: a hue-specific check comparing channel *differences* (`hasWarmAccentPixelInColumns`, `e2e/helpers/pixelDecode.ts` — `r−b` and `r−g` both above a threshold) rather than raw max-min saturation. Differences stay roughly proportional under alpha blending against a roughly-neutral backdrop, so the check still works when the accent color itself renders at partial opacity (a decaying particle) — an exact-color match would not.
- **This sandbox e2e environment's rAF loop runs irregularly under load — real elapsed wait time does not map predictably to audio engine time.** A `waitForTimeout` guess calibrated from one observed run landed a screenshot outside a short-lived effect's ~0.35s active window on a later run, even though the real wait was identical each time — engine time was observed jumping in uneven ~20–90ms increments rather than smooth ~16ms rAF ticks (#484). Poll a live time source (`window.__mawimbi.playback.getEngineTime()`, mawimbi#480's bridge) to a target instead of guessing from wall-clock, and resolve every other geometry-dependent value (bounding rects, sample rate) *before* the poll so the only await between "target reached" and "screenshot taken" is the screenshot call itself. Relatedly: comparing two regions of interest via two separate `page.screenshot()` calls risks each landing at a different real-world instant when the thing under test is itself short-lived — decode **one** screenshot covering both regions (`decodeClip` + a region-scanning check like `hasWarmAccentPixelInColumns`) so both assertions read the same instant.
