# Verification

Catalog of verification patterns that work in this repo — and the ones that look like they work but don't. Every spec's verification design starts by consulting this file (`/spec` step 3) and ends by extending it with anything new that was learned. Mechanics (commands, jsdom limits, e2e environment quirks) live in CLAUDE.md's Testing section; this file is about *choosing* the right verification for a claim.

## Choosing a verification

Work down this list; prefer the earliest level that can actually falsify the claim:

1. **Unit test (vitest)** — service state machines, reducers, pure math (e.g. `runwayConfig.test.ts` sweeps every preset across viewport heights for NaN/Infinity). Read service state via plain getters.
2. **E2E invariant (Playwright)** — anything involving real layout, paint, stacking, scrolling, or cross-feature workflows. Assert *invariants* (alignment within tolerance, element visibility, geometry resync after resize/zoom) rather than pixel snapshots where possible — invariants survive intentional restyling.
3. **Screenshot-decoded pixels** — when even DOM geometry lies. Rect-based assertions pass straight through clipping bugs (rects ignore clipping); decoding actual screenshot pixels at a location is the only automated check that catches a "renders nothing" failure. This class caught the #459 runway clipping bug after rect-based tests stayed green (PR #464).
4. **Visual snapshot (`e2e/__screenshots__/`)** — pins an *intentional look* (per-preset appearance). Costs maintenance on every restyle; use for looks, not logic.
5. **Human QA issue** — what automation cannot see (on-device feel, address-bar viewport behavior, design judgment) gets an explicit checklist issue (pattern: #467). Never leave it silently untested; the escape hatch is a tracked issue, not a shrug.
6. **Profiling issue** — reasoned performance concerns become an issue with a concrete measurement plan and a close-if-comfortable clause (pattern: #469). Don't optimize unmeasured costs; don't silently ignore them either.

## Rules of thumb

- **Bugs get a failing test first** — CLAUDE.md, Bug Fixes. The reproducing test is what defines "fixed".
- **A computed value is not a rendered value** — jsdom's limits are catalogued in CLAUDE.md, Testing; stacking/pseudo-element/paint claims need e2e.
- **Distrust green tests around transforms and clipping.** If a change touches scroll containers, CSS transforms, or z-index, ask what a rect-based test would miss (see #464's visibility invariant).
- **`getBoundingClientRect` lies under 3D transforms** — it returns the axis-aligned bounding box of the *projected trapezoid*, not the logical rectangle, and Playwright's `hover()`/`click()` element-center can project off-viewport at steep tilt. Interact with untransformed wrappers (phantom/viewport) and scan actual canvas buffers instead of doing rect math (PRs #393, #395). Distinct mechanism from "rects ignore clipping" (#464); both mean rect assertions lie near transforms.
- **Reactivity/state-management migrations gate on the e2e suite, not the unit suite.** After the signals migration, all 233 unit tests passed while 0 of 69 e2e tests did (#114): signal effects run outside React rendering, so the audio engine kept responding while the UI silently stopped re-rendering — mock-heavy unit tests cannot see it.
- **Mock the real API shape, not your assumption of it.** Five consecutive classification bugs shipped past a fully green suite because mocks mirrored the code's assumptions about third-party modules (EssentiaWASM export style #272, object-not-factory #273, per-frame arrays #281, ONNX feed input names #284, EffNet's two outputs #295). When wrapping an untyped third-party module, verify the mock against the actual runtime shape (scratch script or e2e smoke) first, and prefer reading dynamic properties (`session.inputNames`, output dims) over hardcoding.
- **Don't assert implementation details.** ~3,000 lines of tests asserting canvas API call counts, worker message formats, node wiring, and RAF/ResizeObserver mock internals were deliberately deleted (PR #365), keeping user scenarios, state-machine transitions, and bug-fix regressions. Several renderers have no unit tests *on purpose* — don't "restore coverage" for their internals.
- **Frequency-mapping changes get a 12-TET test:** any new analysis/visualization path must place the same tone within ±2 bins of existing paths and keep semitone spacing uniform (spacing coefficient of variation < 15%) — the pattern that ended the live/offline misalignment bug class (#220; see kb/domain.md).
- **A spacing/uniformity test must call the function under test, not re-derive its expected output from the same formula the implementation uses.** A first draft of #482's semitone-spacing test computed `midiNoteToBin(midi) / 2` directly instead of feeding synthetic bins through `poolSemitoneBars` and reading the pooled output — it would stay green through an off-by-one or wrong-stride regression in the pooling loop itself, since it never exercised that code. Caught by code review before merge; fixed by pooling synthetic per-note bin arrays and asserting on `indexOf()` in the pooled result.
- **State-flap bugs need transition traces, not visibility polls.** A playback stutter (state flipping every ~250 ms, "playing" for only 1–2 frames per cycle) made `expect(locator).toBeVisible()` false-pass or false-fail depending on where the poll landed — in the same session, a visibility-poll repro test passed while the bug was live. The falsifiable form: inject an rAF sampler via `page.evaluate` that records `{t, title}` *transitions* of the play/pause button, then assert on the transition count (stable playback ⇒ ≤1 transition per user action). Pattern established in spec 002's bug investigation (2026-07-19); delivered as `e2e/helpers/flapTracer.ts` (`tracePlaybackState`/`stopPlaybackTrace`, re-exported via `e2e/fixtures.ts`), exercised by `e2e/playback-toggle.spec.ts` (#473, PR #496). **Corollary (#473):** don't gate the trace's start on a visibility-poll assertion for the state the triggering action produces — during a live flap bug that assertion inherits the same false-fail risk (a real run timed out waiting for `getByTitle('Pause')` because its ~15–40 ms windows fell between polls). Instead seed the trace *before* the triggering action and assert the total transition count allows exactly one (the action's own expected transition), not zero.
- **Flakiness protocol:** re-run the failing spec in isolation before blaming the change; `audio.spec.ts` has flaked under full parallel runs since 2026-03 (PRs #416, #418; reconfirmed in #457's test plan) — a long-lived flake, not a recent one.
- **Verify the fix against a live reproduction when possible** — e.g. PR #465 validated its recovery script against a real `node_modules` wipe, not a simulation.
- **Real Basic Pitch transcription is viable in e2e — measured, not assumed (#480).** Model self-hosted (`public/basic-pitch-model/`), so `fixtures.ts`'s `blockModelRequests` (which blocks `/models/*.onnx`) doesn't touch it. Measured: ~6–11s per upload (cold model load on a run's first upload, ~6–8s steady-state after) including navigation/upload/analysis, and three consecutive runs against the same fixture (`test-tone-short.wav`, a 440 Hz pure tone) produced the identical note (MIDI 69, same start/end time to the millisecond) — deterministic enough to assert on directly. No dev-only injection seam needed; use real transcription for melody-path e2e tests (spec 003's Milestone 4 sparkle tests should reuse this rather than reintroducing the injection-seam fallback).
- **`Tone.Reverb`'s impulse response is generated asynchronously and silently — verified against real Tone 15.1.22 in a browser (#489).** `new Tone.Reverb(...)` kicks off `generate()`, which renders the IR through its *own* nested `OfflineContext` and resolves the `ready` promise when done; until then `ConvolverNode.buffer` is `null`, which is silence, not an error or passthrough. Skipping `await reverb.ready` before starting the source inside a `Tone.Offline` render doesn't fail — it just quietly omits the reverb's contribution, so an assertion on "does the chain produce tail energy" can pass for the wrong reason if another chain member (e.g. `FeedbackDelay`'s echoes) supplies the tail instead. Isolate the effect under test (bypass the others) when asserting on one effect's contribution specifically. Also: `Tone.Filter` has no `wet` (it's a plain `ToneAudioNode`, not an `Effect` subclass) while `Tone.Reverb`/`Tone.FeedbackDelay` do — bypass-via-disconnect (spec 004 Decision 3) is required for `Filter` regardless, and is the uniform choice across all three.
- **Prefer a direct state read over screenshot-decoding when the claim is about data, not paint.** Screenshot-decoded pixels (level 3) exist for when paint itself is the thing being verified — data correctness (did extraction produce the right note?) is better proven by reading the state directly, which is level-1-equivalent even when done through a page-context bridge rather than a plain getter. `window.__mawimbi` (`src/global.d.ts`, wired in `AudioService.ts`, `import.meta.env.DEV`-gated) exists for exactly this: worker-produced state (melody notes) with no DOM/CSS surface a rect or pixel check could read. Reserve pixel-decoding for verifying that state actually rendered (e.g. the sparkle/pulse milestones).
- **A toggle button's e2e assertion races any concurrent async auto-transition — don't click "whatever the button currently says" as a timing-sensitive trigger.** An e2e test proving "an explicit command cancels a stale scheduled auto-resume" by clicking Play then Pause at the ~200ms debounce boundary was itself flaky: if the scheduled auto-resume fires in the same instant the click's handler runs, the toggle's *meaning* (start vs. stop) depends on which happened first, so the identical test intermittently passed or failed under load — a hazard of the toggle, not a bug in the fix it was testing (caught empirically: 4/4 failures after a passing run, traced with a flap-tracer debug spec). Prefer a control whose effect doesn't depend on current state (e.g. Rewind, which always forces stopped+0) to prove a cancellation mechanism deterministically regardless of timing (#475).

- **CDP's `Input.dispatchTouchEvent` can't reliably add a touch to an already-active sequence — reach for a jsdom unit test instead.** Reproducing "a second finger joins mid-gesture without the first lifting" (spec 002 milestone 4, #476) by dispatching a `touchStart` that lists the continuing point (matching its last position, even with an explicit `id`) alongside a new one does fire the native `touchstart` DOM event, but React's synthetic `pointermove`/`pointerdown` events stop firing entirely afterward in this environment — confirmed by instrumenting the real handlers. Don't spend more effort chasing CDP touch-point continuity for this shape of gesture; jsdom's `TouchEvent`/`Touch` constructors work fine for this (verified: `new TouchEvent` and `fireEvent.touchStart(el, { touches: [...] })` both work in this repo's vitest+jsdom setup), so drive the real (unmocked) component with `fireEvent.pointerDown/pointerMove` plus a native `fireEvent.touchStart`/`touchCancel` directly — it exercises the same production code path (`useTimelineZoom`'s real listeners) without depending on CDP's touch synthesis at all.
- **Not every geometry restyle moves `e2e/__screenshots__/`.** Spec 003's #481 (playhead meter: 0.4H→0.25H, centered→bottom-aligned, 2:1→3:1) changed only a thin translucent overlay well under `playwright.config.ts`'s `maxDiffPixelRatio: 0.01`, and the only full-page snapshot (`project-empty.png`) is taken before any track/playhead content exists — `visual.spec.ts` passed unchanged. Run the visual suite before assuming a snapshot update is owed; don't `--update-snapshots` reflexively.
- **Isolating one track's pixels in the "single mixed track" design: use time, not rects.** Every `.timeline__track` shares one CSS grid cell (`kb/domain.md`), so two tracks' wrapping divs have identical bounding rects — no rect-based clip can isolate one track's content from another's. Two tracks of the *same* frequency but *different* duration give a usable window instead: upload the longer one first (background), the shorter one second (active/newest), and sample the on-screen band for elapsed time inside `(shorterDuration, longerDuration)` — only the background track has content there. Force `page.emulateMedia({ reducedMotion: 'reduce' })` first so `scale(s) = 1` everywhere (`kb/domain.md`, Runway geometry) and elapsed time maps *linearly* to screen Y (`playheadLineY − elapsed × pixelsPerSecond`), avoiding the runway's real nonlinear projection. Established in spec 004 milestone 2's background-dimming assertion (`e2e/track-edit-mode.spec.ts`); the existing fixtures (`test-tone-short.wav`/`test-tone-long.wav`) are both 440 Hz, so this only works via duration, not frequency, until a differing-frequency short fixture exists.
- **A paint effect near the runway's far edge can compress to sub-pixel screen height under the real tilted projection — flatten it to test the effect at all.** Verifying the far-edge alpha fade (`FAR_EDGE_FADE_PX = 200` canvas px, spec 006 M4 #541) against the real tilt risked sampling a screen region compressed to a handful of pixels near the horizon — exactly the perspective compression that makes a hard cut there read as invisible in the first place (the whole reason #468 considered "just push the cut toward the horizon" as one fix option). The same `page.emulateMedia({ reducedMotion: 'reduce' })` flattening as the entry above maps canvas bitmap rows 1:1 to screen pixels with no compression, turning the fade band into an ordinary fixed-height region a `meanLuminance` band-sampling loop (`e2e/helpers/pixelDecode.ts`) can measure reliably (`e2e/runway-far-edge.spec.ts`).
- **`magnitudeToByte`'s dB range is `[-80, -30]`, not full-scale dBFS — a synthetic "loud" test magnitude needs a matching dB value or it silently clips.** A `barTransfer.test.ts` (#483) draft picked `magnitude = 1` for a "fundamental" and `10^(-10/20)` for a "harmonic 10 dB weaker", expecting a realistic ratio; both landed above `MAX_DECIBELS = -30` and clamped to byte 255, producing `NaN` (0/0) two steps later rather than a wrong-but-plausible number — the clipping was silent at the point it happened. Pick synthetic dB values from inside the analyser's actual range (e.g. `-35` and `-45`) when constructing CQT-byte test fixtures, and export the constants (`CQTAnalyser.ts` already exports `MIN_DECIBELS`/`MAX_DECIBELS`/`magnitudeToByte` for exactly this) instead of hand-rolling the byte math.
- **`git diff origin/master...HEAD -- <files>` can silently span more than "just the new commit" when a PR merge squashed history that a local branch kept unsquashed.** After PR #516 merged (squash or merge-commit unknown from the branch alone), continuing work on the same local branch left it with the pre-merge multi-commit history still intact, diverged in *shape* but not *content* from the now-squashed `origin/master`. A code-review pass scoped by `git diff origin/master...HEAD -- <changed files>` for "the new commit" actually included two already-merged, already-reviewed commits' content too — a finder agent caught this itself by comparing blob hashes between the intended single-commit diff and the actual reviewed diff. Before scoping a review diff on a branch that outlived a merge, confirm `git diff origin/<default> <branch-tip-before-new-work>` is empty (content-identical) rather than assuming the commit range matches commit *count*.
- **A toolbar toggle button is `pointer-events: none` while its own content sheet is open** (`ToolbarBottomSheet.tsx`'s `toolbar-sheet--hidden`, mirrored by every mixer/lyrics/effects e2e test that closes via `getByTitle('Close')` instead of re-clicking the toggle). Clicking the toggle's title text again to close a sheet hangs for the full test timeout with a `subtree intercepts pointer events` error, not a clear "button not found" — always close through the sheet's own Close button.
- **A translucent overlay's background is not a neutral backdrop — a plain saturation check can't tell its own accent color apart from bleed-through of whatever paints underneath it.** The playhead loudness meter's background renders at alpha 0.15 over the runway, so a track's own (randomized, #21/#36) spectrogram color shows through it. Building the sparkle-burst e2e test (#484), `hasSaturatedPixel` (the #464 pattern) flagged that bleed-through as a false positive for the new sparkle color even with `Math.random` pinned to a fixed track color — both are "saturated", so the check couldn't distinguish them. Fix: a hue-specific check comparing channel *differences* (`hasWarmAccentPixelInColumns`, `e2e/helpers/pixelDecode.ts` — `r−b` and `r−g` both above a threshold) rather than raw max-min saturation. Differences stay roughly proportional under alpha blending against a roughly-neutral backdrop, so the check still works when the accent color itself renders at partial opacity (a decaying particle) — an exact-color match would not.
- **This sandbox e2e environment's rAF loop runs irregularly under load — real elapsed wait time does not map predictably to audio engine time.** A `waitForTimeout` guess calibrated from one observed run landed a screenshot outside a short-lived effect's ~0.35s active window on a later run, even though the real wait was identical each time — engine time was observed jumping in uneven ~20–90ms increments rather than smooth ~16ms rAF ticks (#484). Poll a live time source (`window.__mawimbi.playback.getEngineTime()`, mawimbi#480's bridge) to a target instead of guessing from wall-clock, and resolve every other geometry-dependent value (bounding rects, sample rate) *before* the poll so the only await between "target reached" and "screenshot taken" is the screenshot call itself. Relatedly: comparing two regions of interest via two separate `page.screenshot()` calls risks each landing at a different real-world instant when the thing under test is itself short-lived — decode **one** screenshot covering both regions (`decodeClip` + a region-scanning check like `hasWarmAccentPixelInColumns`) so both assertions read the same instant.
- **Pinning `Math.random` for deterministic e2e color assertions also silences `Tone.Reverb` — it has no other use for randomness in this codebase to warn you.** `Tone.Reverb.generate()` renders its impulse response from white noise seeded by `Math.random()`; pinning it to `() => 0` (the established track-color pattern, #21/#36) makes every IR sample exactly zero, so the reverb becomes completely silent — no error, no warning, just a "tail" that never appears, indistinguishable from the effect simply not being wired up. Found building spec 004 M6's tail-energy e2e test (#494): the assertion window's luminance was bit-identical dry vs. wet only when `Math.random` was pinned. Track colors in this app are not randomized at all (`COLOR_PALETTE[nextColorId]` cycles deterministically, `projectPageReducer.ts`) — don't pin `Math.random` defensively for color determinism unless the test actually exercises a real per-track random assignment; check first.
- **A pixel-window computed from `playheadLineY − elapsed × pixelsPerSecond` can silently land outside the runway's drawn content, sampling inert background instead of erroring.** `Spectrogram.tsx`'s canvas only paints within the scrubber's *canvas window* (`getCanvasWindow`, bounded by `--runway-window-top/bottom`), which at the default zoom (200px/s) covers well under a full viewport-height's worth of elapsed time above the playhead line — often under half of what the naive arithmetic assumes fits on screen. A window request that exceeds it doesn't fail: it just samples real (but content-less) DOM/CSS background, producing a plausible-looking luminance that has nothing to do with the audio (#494). Keep elapsed-time e2e windows short and verify against an actual screenshot (`page.screenshot` to a file, inspected) before trusting a computed clip, rather than assuming the whole area above the playhead is fair game.
- **A same-duration spectrogram refresh needs the tile *array* to invalidate a redraw's dirty check, not the tile *count*.** `Spectrogram.tsx`'s rAF loop only repaints when `lastDrawnRef`'s tracked state differs from the current frame's; spec 004 M6's post-effect refresh (#494) produces a new `tiles` array of the *same length* as the dry one (same track duration), so a `tiles.length === last.tileCount` guard never detects it and the canvas silently keeps showing stale (dry) pixels forever, with no error and every non-visual assertion (analysis ran, cache updated, IndexedDB persisted) green. Comparing the tile array by *reference* (`tiles === last.tiles`) instead of length fixes it for free: `SpectrogramCache` always constructs a fresh array on every `analyse`/`restore`/`setEntry`, so identity alone correctly distinguishes "genuinely new tiles" from "same entry, unrelated re-render." Caught only by watching the real e2e assertion fail with plausible-but-wrong pixel values, not by any unit test (the memo is a rendering-loop concern, not a hook one) — see also `CLAUDE.md`'s note on this pattern.
- **`page.getByTitle('X')` matches any title that *contains* X, case-sensitively but not exact — pass `{ exact: true }` whenever one control's title is a prefix/substring of another's.** Building #523's recording-drawer e2e coverage, `getByTitle('Record')` (the drawer's own control) also matched the toolbar toggle titled `'Hide recording'`/`'Show recording'` (`"Recording"` contains `"Record"`), a strict-mode violation Playwright reports as "resolved to 2 elements" — easy to misread as "the button doesn't exist" rather than "the query is ambiguous." Any two title strings in this app where one is a case-sensitive substring of the other need `{ exact: true }` on the shorter one's locator.
- **`setupTests.ts`'s global Tone mock factories must forward their constructor args, or a constructor-time value silently reads back as the factory's hardcoded default.** `makeGainNode()` (added for spec 005's monitor-gain unit tests, #524) originally ignored its arguments and always returned `gain: makeRampableParam(1)`; `MicrophoneService`'s `new Tone.Gain(initialGainFromVolume)` then read back `gain.value === 1` regardless of what was actually passed in, because `vi.fn().mockImplementation(fn)` calls `fn` with the real constructor arguments but `fn` itself threw them away. The assertion didn't error — it just silently compared against the wrong expected value until traced. Fix: give the factory a parameter with the same default the real Tone node would have (`makeGainNode(initialGain = 1)`) and pass it through. Same class as "mock the real API shape, not your assumption of it" above, but specifically about constructor arguments rather than method/property shape — check both when adding a new mocked Tone node type.
- **A programmatic `MediaStreamTrack.stop()` does not fire the track's `ended` event in Chromium — verified empirically (mawimbi#522).** `ended` only fires when a track stops for a reason *other than* calling `stop()` on it (device unplugged, permission revoked); code that needs to react to a track it controls being stopped must override the track's own `stop()` method instead of listening for `onended`. This bit `e2e/helpers/fakeMicStream.ts`'s cleanup: `MicrophoneService.close()` (via `Tone.UserMedia.close()`) calls `track.stop()` on the fake mic's stream but never tears down the `AudioContext`/`OscillatorNode` backing it (`Tone.UserMedia.close()` only stops tracks, per its source — it has no way to know what created the stream), so a test re-opening the mic (e.g. punch-in/out re-arming, spec 005 milestone 5) would otherwise leak one `AudioContext` per open.
- **`npm run lint` (`eslint src`) and `npm run build` (`tsc -b`, whose `tsconfig.json` `include` is `["src", "vite.config.ts"]`) never touch `e2e/` — a clean lint+build proves nothing about dead code or type errors in e2e specs/helpers.** Building #538's `spectrogram-stats.spec.ts`, an unused constant (`LONG_FIXTURE_SECONDS`, declared then never referenced — the segment values were hardcoded separately) passed `npm run lint && npm run build` with no complaint; only a manual line-by-line self-review (substituting for `/code-review`, which is gated to explicit user invocation per the entry below) caught it. `noUnusedLocals`/`noUnusedParameters` are set in `tsconfig.json` but don't apply outside its `include`, and the `eslint.config.js` flat config's `files: ['**/*.{ts,tsx}']` glob would match `e2e/` if ever invoked over it — but `npm run lint`'s CLI argument (`eslint src`) restricts the actual scan to `src` regardless. Don't trust "lint and build are green" as evidence of a clean e2e diff; read it, or run `npx eslint e2e` / a scoped `tsc --noEmit` over `e2e/` directly if the mistake class recurs.
- **An incompletely-shaped `as unknown as X` mock doesn't fail until new code reads the property it's missing — and by then the failure is a silently wrong computed value, not a type or test error.** `SpectrogramCache.test.ts`'s pre-existing ad hoc tile mocks (`{ close: vi.fn() } as unknown as ImageBitmap`, no `width`/`height`) passed every one of that file's ~20 `analyse()`/`restore()`/`setEntry()` calls for years because nothing read those fields — until #538 added `SpectrogramStats.recordEntry`'s `tile.width * tile.height * 4` byte accounting, which every one of those calls now runs under `import.meta.env.DEV` (true by default under Vitest). `undefined * undefined` is `NaN`, computed and stored silently; no test asserted on the new field, so the suite stayed green with wrong data baked into the cache's own stats. Distinct from "mock the real API shape, not your assumption of it" above (that's about a mock *actively misleading* a test into the wrong expected value) — this is a mock that was adequate for everything it was originally written to support, made inadequate by *later, unrelated* code reading a field it never had. Caught by multi-angle `/code-review` reproducing the computation directly, not by any assertion. When new code starts reading a mocked object's properties for the first time, check whether every existing mock of that type actually has them, not just the ones the new code's own tests introduce.
- **A one-shot "start token" consumed on its first match breaks the moment a caller starts making 2+ deliveries per start.** `SpectrogramStats.recordAnalysisStart`/`recordEntry` (#538) paired one start call with what was assumed to be exactly one matching `recordEntry`, deleting the pending start the first time its token matched. Spec 006 M2's chunked analysis (#539) calls `recordEntry` once per chunk plus once for the final result, all with the *same* token; consuming it on the first (intermediate) delivery left the final, `analysisComplete: true` commit — the one whose timing actually mattered — unable to match, silently reporting `analysisMs: 0` for virtually every real analysis. Caught by `/code-review`, not by any test (none called `recordEntry` twice with the same token). General shape: when a single-use token is threaded through what becomes a multi-delivery flow, gate its consumption on the *terminal* delivery explicitly (an `analysisComplete`-style flag), not "the first delivery that matches" — and a test should specifically exercise "call the consuming function N>1 times with the same token, assert the *last* call gets the real value."
- **A `vi.mock` factory returning a fresh object literal on every call destabilizes any effect that depends on it — and can mask or manufacture test races via incidental timing, not just break things outright.** `useSpectrogramCache.test.ts`'s mocked `useAudioService()` returned `() => ({ spectrogramCache: {...} })`, a brand-new object every call, unlike the real hook (a stable singleton context value). Since the hook's effect deps include `audioService`, every re-render spuriously re-ran the effect, re-reading whatever `getEntry` statically mocked and re-committing it — silently overwriting a *later* state update in a new test exercising a persistent subscription (#539's `subscribeToEntry`). The same instability had been *tolerated* by the file's other tests for years: their assertions raced a background `extractAndCacheMelody` callback's `setEntry({ ...updated })` (a deliberate fresh-object commit, not the literal mocked entry) against `waitFor` polling, and happened to usually catch the brief window before the spurious re-render clobbered it. Stabilizing the mock (hoisting it to a module-level constant instead of a factory literal) flipped that race from "usually passes" to "always fails" for one such test, exposing that its assertion should have used `.toEqual` (content) rather than `.toBe` (reference) all along — a background update can legitimately supersede the literal mocked object with an equal-but-different one. When a mock stands in for a real singleton, make the mock return one too; and audit any `.toBe()` against a mocked entity for async paths that can re-commit an equal-but-different object.

- **A debounced function call only gates *invocation* timing, not the duration of the async work it triggers — two calls 400ms+ apart can still have overlapping in-flight executions.** `EffectsRefreshScheduler`'s `isSuperseded()` guard (kb/decisions.md 2026-07-21) was only checked after the two async stages identified when the pattern was first built; code review (mawimbi#540 follow-up) found the persist-then-notify tail lacked the same guard, and the overlap is reachable whenever one commit's offline-render+analyse+persist outlasts the 400ms debounce window before the next commit's own run fires. General rule for any per-key debounced async scheduler: re-check the supersede/generation guard after *every* await, not just the ones the original design anticipated. To reproduce this class of race in a test, `vi.spyOn` a directly-imported (non-injected) module function — e.g. `vi.spyOn(await import('../../project/ProjectStorageService'), 'saveSpectrogramData')` — and control its resolution order with a manually-held deferred promise for the stale call vs. an immediately-resolving fresh call, then assert the stale call's post-resolution side effects didn't double-fire after the fresh call already completed.
- **This headless sandbox's real `AudioContext` never delivers per-frame `AudioWorkletProcessor` data back to the main thread, for any tap point, regardless of wiring — confirmed empirically, not assumed (#542).** Building the worklet-CQT-for-playback milestone, an e2e assertion expecting the playhead meter to paint brighter frequency bars during playback found no signal at all. Instrumented both sides directly (temporary `console.log` in the main-thread `port.onmessage` handler and inside the actual `AudioWorkletProcessor.process()`/`handleConfigure`, both forwarded to Playwright's `page.on('console', ...)`) and found **zero** messages ever arrive — not just for the new CQT path, but for the pre-existing, already-shipped destination-tapped loudness/RMS meter too, across multiple fixtures and wait durations up to several seconds. This is a sandbox limitation (no real audio output device), not a code defect — the same wiring works in real deployed browsers. Consequence: screenshot-decoded assertions of live worklet-driven visuals (meter bar height scaling with real amplitude, real frequency content) are unfalsifiable in this environment. Verify audio-thread analysis correctness at the unit level instead (mock the `AudioWorkletProcessor`/`AudioWorkletNode` globals — see the next entry — or feed synthetic buffers through the real class), and verify *routing* (which code path actually ran) via bridge counters like `mainThreadCqtConstructions`, not via pixel content that depends on the worklet producing frames it never will here.
- **An `AudioWorkletProcessor` subclass can be unit-tested directly by stubbing its globals with `vi.stubGlobal` before a dynamic import, then driving `.process()`/`.port.onmessage` like any other class.** `AnalysisProcessor.ts` (the audio-thread CQT/RMS processor) was never unit-tested before #542 because `class X extends AudioWorkletProcessor` and the module-level `registerProcessor(...)` call both reference globals that don't exist in jsdom. Fix: `vi.stubGlobal('AudioWorkletProcessor', MockClass)` and `vi.stubGlobal('registerProcessor', vi.fn())` *before* `const { default: X } = await import('../AnalysisProcessor')` (top-level `await` works fine in vitest's ESM setup) — the module then evaluates against the stubs exactly as it would inside a real worklet scope, and `new X()` / `x.process([[chunk]])` / simulating `x.port.onmessage({data: {...}})` exercise the real production code, not a reimplementation of its logic. One type gotcha: TypeScript still resolves `.port` to the real DOM `MessagePort` type (inherited from the declared `extends AudioWorkletProcessor`), which has no `.mock` — cast to the actually-installed mock shape when reaching for `.postMessage.mock.calls`.

- **A fix for a "commit didn't clear state" bug that only addresses the value-changed commit path can look complete against unit tests while still failing a live drag.** Fixing the live-effects-preview overlay's stuck-overlay bug (mawimbi#551) by clearing it directly from `commitAmount` passed its own unit test and looked like a full fix — the same round-trip-drag scenario still failed the instant it was reproduced with a real `page.mouse` drag in the browser, because Radix's `onValueCommit` never fires at all when the released value equals the value at drag-start (see CLAUDE.md, "Radix Slider's `onValueCommit` is not a release event"). The gap was invisible from reading this app's own code — reading `node_modules/@radix-ui/react-slider`'s source directly (`handleSlideEnd`'s `valuesBeforeSlideStartRef` comparison) is what explained the observed behavior and pointed at the actual fix (a pointer-lifecycle handler, independent of any slider value event). General rule, sharpening "verify the fix against a live reproduction when possible" above: for a bug involving a third-party interactive component (drag, pointer capture, keyboard nav), a fix based only on the app's own code and unit-mocked component behavior is unverified until reproduced with a real interaction in a real browser — and when that reproduction contradicts the fix, read the library's actual source for the precise condition, rather than guessing at a second patch.

- **A zero-committed-code browser evaluation harness is possible because the Vite dev server serves `/node_modules/<pkg>/...` and `/src/**/*.ts` (transpiled on the fly) directly to any dynamic `import()` — but two gotchas bite when using it from a scratch e2e spec.** Building spec 007 M1's library-evaluation harness (#557), a `page.evaluate()` block could `await import('/src/features/spectrogram/CQTAnalyser.ts')` to run the app's *real*, unmodified analysis code, and `await import('/node_modules/signalsmith-stretch/SignalsmithStretch.mjs')` to load a third-party package — both confirmed servable by `curl`ing the dev server directly before trusting them in a test. First gotcha: a raw `node_modules` path bypasses Vite's dependency **pre-bundling** (the `.vite/deps/` cache, which shims CJS→ESM interop for transitive deps) — importing `tone`'s real entry this way threw `SyntaxError: ... does not provide an export named 'AutomationEventList'` because a transitive CJS dep (`automation-events`) needs that shim; importing `/node_modules/.vite/deps/tone.js` instead (the same URL the app's own bundled code resolves to) worked. Second gotcha: a module `Worker` constructed from a `Blob` URL (the standard way to spin up worker code with nothing committed to disk) cannot resolve **root-relative** (`/...`) dynamic-import specifiers from inside the worker — `Failed to resolve module specifier '/node_modules/...'` — even though the identical path resolves fine from the main-thread page context; only a fully-qualified URL (`` `${location.origin}/node_modules/...` ``) works from inside the blob worker. Neither gotcha is specific to this repo's code — they're general Chromium/Vite behaviors worth knowing before assuming a "package not found" or "export missing" error means the package itself is broken.
- **A near-zero-vs-low confidence output from a real (non-mocked) third-party algorithm can be a fixture-duration artifact, not a signal-quality one — check by scaling the input before trusting the number.** Spec 007 M1's essentia `RhythmExtractor2013` shape validation (#557) got `confidence: 0` (not merely low — exactly zero) from both the `multifeature` and `degara` methods on an early ~5 s/8-beat click fixture, even though the BPM estimate itself was accurate (119.84 vs. 120 ground truth) — a result that could easily be misread as "this method just doesn't produce confidence for synthetic clicks." Tiling the identical waveform to ~20 s (no change in click content, spacing, or amplitude) produced a normal-range confidence (3.4–3.8, essentia's typical 0–5.32 scale) with no change to BPM accuracy — proving it was the estimator's minimum-duration requirement, not the signal. General shape: when a real (not mocked) library call returns a suspiciously degenerate value (exactly 0, exactly 1, always the same class) on a minimal synthetic fixture, testing the same fixture scaled up (longer, louder, more repetitions) before concluding "this is how the library behaves" is cheap and can save a wrong assumption from reaching a mock or a fixture that ships too short to ever be useful.
- **`SignalsmithStretch`'s (and by the same pattern, other AudioWorklet-node npm packages') worklet-module registration is self-contained — no static-asset wiring or `addModule` call needed from the app.** `SignalsmithStretch(audioContext, options)` first tries `new AudioWorkletNode(ctx, key, options)`; if that throws (module not yet registered on this context), it builds the *same already-loaded module's own source* into a `Blob` and calls `ctx.audioWorklet.addModule(URL.createObjectURL(blob))` itself, then retries the constructor — confirmed by reading `node_modules/signalsmith-stretch/SignalsmithStretch.mjs` directly (spec 007 M1, #557), not assumed from the README. This is also why it works identically on a fresh `OfflineAudioContext`/`Tone.OfflineContext` per render call (`renderTrackOffline.ts`'s pattern rebuilds the whole chain from scratch every time) with no caching or pre-registration step required from calling code.
