# Verification

Catalog of verification patterns that work in this repo — and the ones that look like they work but don't. Every spec's verification design starts by consulting this file (`/spec` step 3) and ends by extending it with anything new that was learned. Mechanics (commands, jsdom limits, e2e environment quirks) live in CLAUDE.md's Testing section; this file is about *choosing* the right verification for a claim.

## Choosing a verification

Work down this list; prefer the earliest level that can actually falsify the claim:

1. **Unit test (vitest)** — service state machines, reducers, pure math (e.g. `runwayConfig.test.ts` sweeps every preset across viewport heights for NaN/Infinity). Read service state via plain getters.
2. **E2E invariant (Playwright)** — anything involving real layout, paint, stacking, scrolling, or cross-feature workflows. Assert *invariants* (alignment within tolerance, element visibility, geometry resync after resize/zoom) rather than pixel snapshots where possible — invariants survive intentional restyling.
3. **Screenshot-decoded pixels** — when even DOM geometry lies. Rect-based assertions pass straight through clipping bugs (rects ignore clipping); decoding actual screenshot pixels at a location is the only automated check that catches a "renders nothing" failure. This class caught the #459 runway clipping bug after rect-based tests stayed green (PR #464).
4. **Visual snapshot (`e2e/__screenshots__/`)** — pins an *intentional look* (per-preset appearance). Costs maintenance on every restyle; use for looks, not logic.
5. **Human QA issue** — what automation cannot see (on-device feel, address-bar viewport behavior, design judgment) gets an explicit checklist issue (pattern: #467). Never leave it silently untested; the escape hatch is a tracked issue, not a shrug.
6. **Profiling issue** — reasoned performance concerns become an issue with a concrete measurement plan and a close-if-comfortable clause (pattern: #469). Don't optimize unmeasured costs; don't silently ignore them either.

## Rules of thumb

- **Bugs get a failing test first** — CLAUDE.md, Bug Fixes. The reproducing test is what defines "fixed".
- **A computed value is not a rendered value** — jsdom's limits are catalogued in CLAUDE.md, Testing; stacking/pseudo-element/paint claims need e2e.
- **Distrust green tests around transforms and clipping.** If a change touches scroll containers, CSS transforms, or z-index, ask what a rect-based test would miss (see #464's visibility invariant).
- **`getBoundingClientRect` lies under 3D transforms** — it returns the axis-aligned bounding box of the *projected trapezoid*, not the logical rectangle, and Playwright's `hover()`/`click()` element-center can project off-viewport at steep tilt. Interact with untransformed wrappers (phantom/viewport) and scan actual canvas buffers instead of doing rect math (PRs #393, #395). Distinct mechanism from "rects ignore clipping" (#464); both mean rect assertions lie near transforms.
- **Reactivity/state-management migrations gate on the e2e suite, not the unit suite.** After the signals migration, all 233 unit tests passed while 0 of 69 e2e tests did (#114): signal effects run outside React rendering, so the audio engine kept responding while the UI silently stopped re-rendering — mock-heavy unit tests cannot see it.
- **Mock the real API shape, not your assumption of it.** Five consecutive classification bugs shipped past a fully green suite because mocks mirrored the code's assumptions about third-party modules (EssentiaWASM export style #272, object-not-factory #273, per-frame arrays #281, ONNX feed input names #284, EffNet's two outputs #295). When wrapping an untyped third-party module, verify the mock against the actual runtime shape (scratch script or e2e smoke) first, and prefer reading dynamic properties (`session.inputNames`, output dims) over hardcoding.
- **Don't assert implementation details.** ~3,000 lines of tests asserting canvas API call counts, worker message formats, node wiring, and RAF/ResizeObserver mock internals were deliberately deleted (PR #365), keeping user scenarios, state-machine transitions, and bug-fix regressions. Several renderers have no unit tests *on purpose* — don't "restore coverage" for their internals.
- **Frequency-mapping changes get a 12-TET test:** any new analysis/visualization path must place the same tone within ±2 bins of existing paths and keep semitone spacing uniform (spacing coefficient of variation < 15%) — the pattern that ended the live/offline misalignment bug class (#220; see kb/domain.md).
- **State-flap bugs need transition traces, not visibility polls.** A playback stutter (state flipping every ~250 ms, "playing" for only 1–2 frames per cycle) made `expect(locator).toBeVisible()` false-pass or false-fail depending on where the poll landed — in the same session, a visibility-poll repro test passed while the bug was live. The falsifiable form: inject an rAF sampler via `page.evaluate` that records `{t, title}` *transitions* of the play/pause button, then assert on the transition count (stable playback ⇒ ≤1 transition per user action). Pattern established in spec 002's bug investigation (2026-07-19). **Corollary (#473):** don't gate the trace's start on a visibility-poll assertion for the state the triggering action produces — during a live flap bug that assertion inherits the same false-fail risk (a real run timed out waiting for `getByTitle('Pause')` because its ~15–40 ms windows fell between polls). Instead seed the trace *before* the triggering action and assert the total transition count allows exactly one (the action's own expected transition), not zero.
- **Flakiness protocol:** re-run the failing spec in isolation before blaming the change; `audio.spec.ts` has flaked under full parallel runs since 2026-03 (PRs #416, #418; reconfirmed in #457's test plan) — a long-lived flake, not a recent one.
- **Verify the fix against a live reproduction when possible** — e.g. PR #465 validated its recovery script against a real `node_modules` wipe, not a simulation.
- **A toggle button's e2e assertion races any concurrent async auto-transition — don't click "whatever the button currently says" as a timing-sensitive trigger.** An e2e test proving "an explicit command cancels a stale scheduled auto-resume" by clicking Play then Pause at the ~200ms debounce boundary was itself flaky: if the scheduled auto-resume fires in the same instant the click's handler runs, the toggle's *meaning* (start vs. stop) depends on which happened first, so the identical test intermittently passed or failed under load — a hazard of the toggle, not a bug in the fix it was testing (caught empirically: 4/4 failures after a passing run, traced with a flap-tracer debug spec). Prefer a control whose effect doesn't depend on current state (e.g. Rewind, which always forces stopped+0) to prove a cancellation mechanism deterministically regardless of timing (#475).
