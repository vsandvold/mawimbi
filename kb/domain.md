# Domain

Audio/music domain knowledge that explains *why* the features work the way they do. Architecture and code layout live in CLAUDE.md; this file is the domain reasoning behind them.

## Analysis and visualization

- **CQT (constant-Q transform), not linear FFT**, drives the spectrograms: constant-Q spacing is logarithmic in frequency, so each octave gets equal visual height and musical pitch maps to consistent vertical distance. That is what makes the "see sounds like you hear them" promise hold. (`features/spectrogram/`)
- **CQT is the third algorithm; each predecessor died structurally, not by tuning.** Single FFT: unusable bass resolution. Dual-band FFT (split ~752 Hz, #194): a 17× temporal/frequency-resolution cliff at the split — a drum hit's bass smeared over ~400 ms while its highs were crisp 23 ms events (#232). 4-band STFT (#234): smoothed adjacent-band jumps to ≤4×, but the 17× low-to-high disparity is inherent to STFT time-frequency uncertainty — more bands cannot fix it (#236). Also evaluated and rejected: Gaborator/WASM CQT, superlets, adaptive windowing, pitch-shift-before-FFT (#232, #233). CQT eliminates seams by construction (per-bin window lengths, uniform Q≈34.1, 24 bins/octave from C1). (#239)
- **CQT dissolved two old problems for free:** pure-TS CQT + pure-JS worklet FFT removed every `OfflineAudioContext.suspend`/`ScriptProcessorNode` dependency, ending the 2020 Safari incompatibility (#27, #190 closed superseded) and mooting the planned mel-scale mapping (#159) — `FUTURE_PLANS.md` §1a–1c predate this.
- **CQT kernels are precomputed per sample rate and capped.** essentia's `SpectrumCQ` was rejected because it recomputes kernels on every call (~433 ms/frame, #236). Uncapped low-frequency kernels (~1 s at 32.7 Hz) smear transients into cone-shaped artifacts, so kernels are capped at `MAX_KERNEL_HOPS = 4` — reduced frequency resolution below ~340 Hz, traded deliberately for visual fidelity (#242).
- **Every analysis path must share the CQT bin definition.** Independently computed frequency→pixel mappings shipped four misalignment bugs (#197, #218, #220, #230); the guard is a 12-TET spacing test (kb/verification.md).
- **Pool bins with max, never sum:** summing overflowed `Uint8Array` values mod 256 (corrupted spectrogram), and max also preserves the strongest component per band (#152, #195).
- **Real-time loudness visuals use RMS** (`Tone.Meter`) with a 0.6 power curve for perceptual linearity, driving a CSS custom property each frame outside React: peak is too jumpy for a smooth visual, LUFS needs a K-weighting chain and is meant for normalization compliance, not 60 fps animation (#98, #101).
- **Two analysis paths** exist because their latency budgets differ: offline CQT analysis renders uploaded audio into cached tiles (`OfflineAnalyser`, `SpectrogramCache`); live worklet analysis (`LiveCQTAnalyser`, `AnalysisProcessor`) renders the in-progress recording frame-by-frame (~25ms hop). When recording completes, the rough live rendering is replaced by the polished offline result — the same live-then-refine pattern is planned for pitch (issue #302).

## ML pipelines

All inference runs in Web Workers (melody: Spotify Basic Pitch; lyrics: Whisper; classification: the chain below).

- **Classification chain:** essentia WASM mel spectrogram (`TensorflowInputMusiCNN`, 96 bands) → Discogs-EffNet embeddings (1280-dim) → *two* ONNX heads sharing that one embedding pass: the MTG-Jamendo instrument head, plus a voice/instrumental gate that overrides the instrument label when voice is detected — the instrument head misclassifies vocal stems as drums/bass/synth (#308).
- **`TensorflowInputMusiCNN` already applies `log10(1 + 10000·x)` compression.** Applying the (spec-mandated, #268) compression a second time crushed all spectral features and classified everything as "synthesizer" (#305). When a feature-extraction spec names a transform, check whether the library layer already performs it.
- **Preprocessing trims silence, doesn't window:** leading/trailing silence is trimmed (10 ms RMS windows) and the full signal fed — fixed-window "loudest segment" extraction was tried and dropped twice (#261, #275→#277). Audio shorter than one EffNet patch (~2.07 s at 16 kHz) returns `unknown` without loading any model (#275).
- **Whisper word timestamps require the `_timestamped` export.** Word timing comes from cross-attention decoder outputs that standard ONNX exports omit (runtime error: "Model outputs must contain cross attentions"). The app uses `onnx-community/whisper-base_timestamped`; any model-size upgrade must pick a `_timestamped` variant (#345). Client-side whisper-base (~142 MB) is the size/quality point that preserves the no-backend rule (#319).

## Timing and mixing

- **`Tone.Transport` is the single timeline clock.** All track playback, the playhead, and recording alignment derive from it; never introduce a second clock.
- **Loudness normalization on upload** (`LoudnessNormalizer`) exists so tracks recorded/uploaded at wildly different levels mix at comparable loudness without the user touching faders first — an amateur-friendliness rule, not an audio-engineering nicety. Design: targets RMS 0.2 (≈ −14 dBFS, the common streaming loudness target), skips near-silent tracks (RMS < 0.001), and applies the gain *inside the channel* on top of the slider's dB value — the slider keeps its full range and the derived initial position restores original perceived loudness on load (#119).
- **Latency compensation** (`features/recording/LatencyCompensation`): a recorded overdub is captured late relative to what the user heard. Compensation = `outputLatency + baseLatency + lookAhead +` a one-render-quantum (128-sample) input estimate (#209), applied by *trimming the recorded buffer* (`trimBuffer`) rather than passing an offset downstream — the offset-parameter design silently lost the offset in undo/redo's `recreateChannel` (#212). A manual click-track calibration tool was deliberately not built (#170). Background research: `FUTURE_PLANS.md`.
- **Scrub-to-seek contract** (#19 → #89–#94): scrubbing pauses playback, seeks debounced (~200 ms), and auto-resumes only if playback was active before the scrub. Resuming must never re-seek — the transport seeks only on an explicit pending seek, otherwise it continues from its own position (a stale-scroll re-seek once jumped playback backwards, #94; formalized as the pending-seek mechanism in #107).

## Audio engine

- Tone.js builds on `standardized-audio-context`, which wraps the native `AudioContext` in a proxy that tracks every node it creates in an internal registry. `Tone.getContext().rawContext` returns that **wrapper**, so connecting a wrapper-created node to a native node fails the registry lookup and throws "value with the given key could not be found" ([Tone.js #712](https://github.com/Tonejs/Tone.js/issues/712)). This is why native-node code paths must build their *entire* chain on the extracted native context — the actionable pattern lives in CLAUDE.md ("Tone.js context is not the native AudioContext"). The workaround survived the Tone v14→v15 upgrade unchanged (v15 still builds on standardized-audio-context; the upgrade needed only `Tone.context` → `Tone.getContext()`, #404).
- **Same family — global-singleton staleness:** the static `Tone.Transport` export binds to the context at import time, so after `Tone.setContext()` it still drives the *old* context's transport — timeline scrolls, no sound (#133; always `Tone.getTransport()`). Any node created *before* `setContext()` lands on the default context, which `Transport.start()` never resumes — the recorder captured silence (#134). Since #135, `AudioService` resolves transport/context once at construction and everything goes through those fields.
- **`decodeAudioData` detaches the `ArrayBuffer` passed to it.** Persisting that buffer afterward silently writes empty bytes — uploads appeared saved to IndexedDB but restored as nothing (#260). Clone the buffer before decoding when the raw bytes are also stored.
- **`Tone.UserMedia.open()` hardcodes its getUserMedia constraints** with no override API. Low-latency capture calls `navigator.mediaDevices.getUserMedia()` directly (echo cancellation/noise suppression/AGC off, mono, `latency: 0`) and injects the stream into UserMedia's private `_stream`/`_mediaStream` fields (#219). Fragile across Tone upgrades — recheck on any Tone bump.

## Runway geometry (playhead/scrubber)

Three physical facts govern all runway work; the decisions built on them are in `kb/decisions.md` (2026-07-18/19 entries):

- Under perspective, the layout→screen mapping is **nonlinear** — any linear-fraction shortcut (CSS `calc`, `cqh` padding) looks right in the flat case and drifts under real tilt.
- Scroll clipping happens in **pre-transform layout space** — a scroll container inside the tilt clips content before `rotateX` can project it into view (the #459 bug class).
- **2D child transforms compose with the plane's tilt; per-child depth does not.** A `.timeline__track`'s own scale/translate/filter renders fine inside the flattened `rotateX` plane, but giving a child its *own* `translateZ` requires `transform-style: preserve-3d` on every intermediate ancestor (any `overflow`/`opacity<1`/`filter` on the chain silently flattens it), and the lifted content then un-anchors from the playhead line unless a compensating translate is solved via `runwayProjection` (fact 1). This is why spec 004 phases per-track separation: in-plane emphasis first, true 3D lift last.
